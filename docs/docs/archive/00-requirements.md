# Требования к системе хранения данных

## 1. Бизнес-контекст

Система предназначена для хранения видео, звука и других данных архива, организованными в виде **фблоков**. Каждый фблок — это самодостаточная единица данных, которая может храниться на диске, а также (вне нашей ответственности) передаваться по сети или обрабатываться независимо от других фблоков.

**Ключевая идея:** фблок — это логическая единица работы приложения фиксированного размера. Фблок содержит данные архива, актуальные за промещуток времени. Система обеспечивает храниение этих данных в течении промежутка времени (N дней).

## 2. Терминология

| Термин | Описание |
|--------|----------|
| **Фблок (fblock)** | Логическая единица данных фиксированного размера. Самодостаточен, не зависит от других фблоков. |
| **Фчанк (fchunk)** | Часть фблока, минимальная единица для операций ввода-вывода и проверки записи. Внутренняя деталь реализации. |
| **Битый фчанк** | Фчанк с физическим дефектом, запись в который невозможна. |
| **Битый фблок** | Фблок, содержащий хотя бы один битый фчанк. Считается непригодным. |
| **Защищённый фблок** | Фблок, помеченный потребителем как read-only. |
| **Потребитель** | Процесс, использующий библиотеку. Заполняет фблоки, запрашивает чтение, работает с метриками и уведомлениями. Может управлять несколькими экземплярами библиотеки. |
| **Хранилище (storage)** | Место хранения данных: файл или блочное устройство. |
| **Время начала актуальности** | Момент, с которого данные считаются актуальными. Задаётся потребителем при заполнении. |
| **Время окончания актуальности** | Момент, после которого данные считаются неактуальными. Задаётся потребителем. Основание для перезаписи. |
| **Время записи** | Момент начала записи фблока на диск. Заполняется библиотекой. |

## 3. Требования к фблоку как самодостаточной единице

### 3.1. Независимость
- Фблок содержит всю информацию, необходимую для интерпретации своих данных.
- Для чтения фблока не требуются метаданные других фблоков или глобальный заголовок хранилища.

### 3.2. Идентификация

| Идентификатор | Назначение | Кто использует |
|---------------|------------|----------------|
| **Индекс фблока** | Физический адрес для циклической записи (0..N-1) | Внутренняя реализация |
| **UUID фблока** | Логический идентификатор, уникальный навсегда | Потребитель |

- UUID генерируется библиотекой **до записи** фблока в хранилище.
- UUID хранится в индексах хранилища.
- Формат: UUIDv4 (128 бит).

### 3.3. Инварианты
- Размер фблока фиксирован для всего хранилища.

## 4. Функциональные требования

### 4.1. Инициализация хранилища

Библиотека должна уметь:
- Создать файл нужного размера и зарезервировать место (для файлового режима).
- Подготовить структуры метаданных к работе.
- Запретить повторную инициализацию без явного подтверждения.

После инициализации хранилище готово к записи и чтению.

### 4.2. Запись фблока

1. **Выбор места:** система движется по кругу, выбирая первый фблок, который:
   - Не битый.
   - Не защищённый.
   - С истекшим временем окончания актуальности.

2. Если подходящий фблок не найден — авария «нет свободного места».

3. Запись выполняется пофчанково с проверкой (write-verify):
   - При обнаружении битого фчанка весь фблок помечается битым.
   - Запись переносится на следующий подходящий фблок.

4. После успешной записи:
   - Потребитель получает уведомление с UUID об успешной записи.

**Важно:** частичная запись недопустима — фблок либо записан целиком, либо не записан.

### 4.3. Чтение фблока

- Поддержка частичного чтения: любой непрерывный диапазон внутри фблока.
- Чтение не привязано к размеру фчанка.

### 4.4. Приоритеты операций

- Запись имеет абсолютный приоритет над чтением.
- Чтение выполняется в паузах между записями.

### 4.5. Управление состоянием фблока

| Состояние | Описание | Перезапись |
|-----------|----------|------------|
| Свободен | Нет данных | Разрешена |
| Занят (актуален) | Есть данные, срок актуальности не истёк | Запрещена |
| Занят (неактуален) | Есть данные, срок актуальности истёк | Разрешена |
| Защищён пользователем | Read-only | Запрещена |
| Битый | Физическое повреждение | Запрещена |

### 4.6. Защита от записи

- Пометка фблока как защищённого в индексах хранилища.
- Защита сохраняется в метаданных и переживает перезапуск.
- Защита не снимается автоматически ни по истечении времени актуальности, ни при каких-либо других условиях. Только явным действием потребителя.

### 4.7. Очередь записи и backpressure

- В памяти поддерживается пул буферов фиксированного размера.
- Готовые к записи фблоки ставятся в очередь FIFO.
- При переполнении очереди библиотека сигнализирует потребителю о перегрузке (backpressure).
- Библиотека **не отбрасывает** данные и **не анализирует** их важность — только сигнализирует.
- Потребитель сам решает, какие данные перестать записывать в фблоки.

### 4.8. Запись

Система должна обеспечить хранение фблоков N дней. Для этого при выборе фблока для перезаписи, смотрит на время актуальности. При отстутсвии фблоков для перезаписи, система сигнализирует об закончившемся месте.

1. **Получение буфера:** потребитель запрашивает буфер для заполнения. Ответ может содержать статус загрузки системы (норма, предупреждение, перегрузка, нет буферов).

2. **Передача заполненного буфера:** потребитель возвращает буфер с данными и метаданными (время начала и окончания актуальности).

### 4.9. Уведомления

#### О новом фблоке
- Отправляется до успешной записи.
- Содержит UUID.

#### О завершении нового фблоке
- Отправляется после успешной записи.
- Содержит UUID.

#### Об удалении фблока
- Отправляется перед перезаписью старого фблока.
- Содержит UUID удаляемого фблока.

### 4.10. Консистентность при сбоях

- При сбое питания фблок может остаться в незавершённом состоянии.
- В фоне система обнаруживает такие фблоки и помечает их как битые.
- Восстановление данных — отдельная CLI-утилита.

### 4.11. Мониторинг состояния диска

- Отслеживание количества битых фблоков.
- Порог: 5% (настраивается).
- При превышении — алерт, работа продолжается.

## 5. Нефункциональные требования

### 5.1. Масштабируемость
- Хранилища до 20 ТБ и более.
- Количество фблоков: до нескольких миллионов.
- Требуется возможность расширения хранилища.

### 5.2. Производительность
- Запись с проверкой упирается в скорость диска.
- Частичное чтение без лишних операций ввода-вывода.

### 5.3. Надёжность
- Контрольные суммы для критических данных.
- Копии критических метаданных в разных зонах диска.

### 5.4. Переносимость
- Фблок, созданный на одной платформе, читается на другой.
- Формат не зависит от файловой системы.

## 6. Ключевые архитектурные решения

| Вопрос | Решение |
|--------|---------|
| Размер фблока | Фиксируется при инициализации |
| Размер фчанка | 4–16 МБ |
| Порог битости фблока | Один битый фчанк = весь фблок битый |
| Идентификация | UUIDv4, генерируется до записи, хранится в индексах хранилища |
| Циклическая запись | Пропуск битых, защищённых и актуальных фблоков |
| Метрики | Сбор в памяти, доступ через API, без хранения истории |
| Восстановление | Отдельная CLI-программа, работает независимо |

## 7. Восстановление данных (требования к CLI-утилите)

- Чтение всего хранилища, включая повреждённые области.
- Извлечение фблоков по UUID.
- Возможность полного или частичного восстановления фблока.


## 8. Импорт
Нужна возможность импортировать другое хранилища.
