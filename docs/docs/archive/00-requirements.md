# Требования к системе хранения данных

## 1. Бизнес-контекст

Система предназначена для хранения видео, звука и других данных архива, организованных в виде **фблоков**. Каждый фблок — это самодостаточная единица данных, которая может храниться на диске, а также (вне нашей ответственности) передаваться по сети или обрабатываться независимо от других фблоков.

**Ключевая идея:** фблок — это логическая единица работы приложения фиксированного размера. Фблок содержит данные архива, актуальные за промежуток времени. Система обеспечивает хранение этих данных в течение заданного срока (N дней).

## 2. Терминология

| Термин | Описание |
|--------|----------|
| **Фблок (fblock)** | Логическая единица данных фиксированного размера. Самодостаточен, не зависит от других фблоков. |
| **Фчанк (fchunk)** | Часть фблока, единица операций записи с верификацией (write-verify). Внутренняя деталь реализации. |
| **Порция чтения** | Небольшой фрагмент данных, читаемый за один системный вызов. Размер выбирается меньше фчанка, чтобы не блокировать запись (см. 4.4). Внутренняя деталь реализации. |
| **Битый фчанк** | Фчанк с физическим дефектом, запись в который невозможна. |
| **Битый фблок** | Фблок, содержащий хотя бы один битый фчанк. Считается непригодным. |
| **Защищённый фблок** | Фблок, помеченный потребителем как read-only. |
| **Потребитель** | Процесс, использующий библиотеку. Заполняет фблоки, запрашивает чтение, работает с метриками и уведомлениями. Один потребитель может работать с несколькими независимыми хранилищами одновременно. |
| **Хранилище (storage)** | Место хранения данных: файл или блочное устройство. |
| **Время начала актуальности** | Момент, с которого данные считаются актуальными. Задаётся потребителем при заполнении. |
| **Время окончания актуальности** | Момент, после которого данные считаются неактуальными. Задаётся потребителем. Основание для перезаписи. |
| **Время записи** | Момент начала записи фблока на диск. Заполняется библиотекой. |

## 3. Требования к фблоку как самодостаточной единице

### 3.1. Независимость
- Фблок содержит всю информацию, необходимую для интерпретации своих данных.
- Для чтения фблока не требуются метаданные других фблоков или глобальный заголовок хранилища.

### 3.2. Идентификация

| Идентификатор | Назначение | Кто использует |
|---------------|------------|----------------|
| **Индекс фблока** | Физический адрес для циклической записи (0..N-1) | Внутренняя реализация |
| **UUID фблока** | Логический идентификатор, уникальный навсегда | Потребитель |

- UUID генерируется библиотекой **до записи** фблока в хранилище.
- UUID хранится в индексах хранилища.
- Формат: UUIDv4 (128 бит).

### 3.3. Инварианты
- Размер фблока фиксирован при инициализации хранилища и не меняется в течение всей его жизни, включая операции расширения и сужения.
- Индекс фблока лежит в диапазоне `0..N-1`, где `N` — текущее количество фблоков в хранилище.
- UUID каждого записанного фблока уникален в пределах хранилища и не переиспользуется.

## 4. Функциональные требования

### 4.1. Инициализация хранилища

Библиотека должна уметь:
- Создать файл нужного размера и зарезервировать место (для файлового режима).
- Подготовить структуры метаданных к работе.
- Запретить повторную инициализацию без явного подтверждения.

После инициализации хранилище готово к записи и чтению.

### 4.2. Запись фблока

1. **Выбор места:** система движется по кругу, выбирая первый фблок, который:
   - Не битый.
   - Не защищённый.
   - С истекшим временем окончания актуальности.

2. Если подходящий фблок не найден — авария «нет свободного места».

3. Запись выполняется пофчанково с проверкой (write-verify):
   - При обнаружении битого фчанка весь фблок помечается битым.
   - Запись переносится на следующий подходящий фблок.

4. После успешной записи:
   - Потребитель получает уведомление с UUID об успешной записи.

**Важно:** частичная запись недопустима — фблок либо записан целиком, либо не записан.

### 4.3. Чтение фблока

- Поддержка частичного чтения: любой непрерывный диапазон внутри фблока.
- Чтение не привязано к размеру фчанка.
- Внутренне одно логическое чтение разбивается на несколько порций чтения, чтобы не блокировать запись на длительное время (см. 4.4).

### 4.4. Приоритеты операций

- Запись имеет абсолютный приоритет над чтением.
- Чтение выполняется в паузах между записями.
- Между порциями чтения система проверяет очередь записи и при наличии готового к записи фблока прерывает чтение до завершения записи.

### 4.5. Управление состоянием фблока

| Состояние | Описание | Перезапись |
|-----------|----------|------------|
| Свободен | Нет данных | Разрешена |
| Занят (актуален) | Есть данные, срок актуальности не истёк | Запрещена |
| Занят (неактуален) | Есть данные, срок актуальности истёк | Разрешена |
| Защищён пользователем | Read-only | Запрещена |
| Битый | Физическое повреждение | Запрещена |

### 4.6. Защита от записи

- Пометка фблока как защищённого в индексах хранилища.
- Защита сохраняется в метаданных и переживает перезапуск.
- Защита не снимается автоматически ни по истечении времени актуальности, ни при каких-либо других условиях. Только явным действием потребителя.
- Библиотека не ограничивает количество защищённых фблоков. Контроль разумной доли защищённых фблоков — ответственность потребителя. Потребитель может получить метрику доли защищённых фблоков через API.

### 4.7. Очередь записи и backpressure

- В памяти поддерживается пул буферов фиксированного размера.
- Готовые к записи фблоки ставятся в очередь FIFO.
- При переполнении очереди библиотека сигнализирует потребителю о перегрузке (backpressure).
- Библиотека **не отбрасывает** данные и **не анализирует** их важность — только сигнализирует.
- Потребитель сам решает, какие данные перестать записывать в фблоки.

### 4.8. API заполнения фблока

Цикл работы потребителя при записи:

1. **Получение буфера:** потребитель запрашивает буфер для заполнения. Ответ содержит буфер и текущий статус загрузки системы (норма, предупреждение, перегрузка, нет буферов).

2. **Заполнение буфера:** потребитель записывает данные в буфер произвольным образом.

3. **Передача заполненного буфера:** потребитель возвращает буфер с метаданными (время начала и окончания актуальности). Буфер ставится в очередь записи.

Решение о том, какой фблок будет перезаписан, принимается библиотекой в момент выполнения записи из очереди (см. 4.2), а не в момент получения буфера.

### 4.9. Уведомления

Все уведомления асинхронны и содержат UUID соответствующего фблока.

| Уведомление | Момент отправки |
|-------------|-----------------|
| **О начале записи фблока** | Перед первым обращением к диску для записи данных фблока |
| **О завершении записи фблока** | После успешной записи и верификации всех фчанков |
| **Об удалении фблока** | Перед перезаписью старого фблока новыми данными |

Уведомление о начале записи позволяет потребителю зарезервировать UUID до того, как данные окажутся на диске.

### 4.10. Консистентность при сбоях

- При сбое питания фблок может остаться в незавершённом состоянии (часть фчанков записана, часть — нет).
- Незавершённость определяется по признакам целостности, заложенным в формат фблока (контрольные суммы, маркеры конца записи — конкретика в документе формата хранилища).
- В фоне система сканирует хранилище и помечает обнаруженные незавершённые фблоки как битые.
- Восстановление данных из частично записанных и битых фблоков выполняется отдельной CLI-утилитой (см. раздел 7).

### 4.11. Мониторинг состояния диска

- Отслеживание доли битых фблоков от общего числа фблоков в хранилище.
- Порог: 5% (настраивается при инициализации).
- При превышении порога — алерт потребителю, работа продолжается.
- Отдельно отслеживается доля защищённых фблоков (без жёсткого порога, для информирования потребителя).

## 5. Нефункциональные требования

### 5.1. Масштабируемость
- Хранилища до 20 ТБ и более.
- Количество фблоков: до нескольких миллионов.
- Поддержка расширения и сужения хранилища без перезаписи существующих фблоков (см. 4.12).

### 5.2. Производительность
- Запись с проверкой упирается в скорость диска.
- Частичное чтение без лишних операций ввода-вывода.
- Размер порции чтения выбирается так, чтобы задержка прерывания чтения для освобождения диска под запись не превышала разумного значения (конкретика — в документе операций).

### 5.3. Надёжность
- Контрольные суммы для критических данных (заголовок фблока, индексы, каждый фчанк).
- Копии критических метаданных (индексов) распределены по хранилищу (см. ADR-002).

### 5.4. Переносимость
- Фблок, созданный на одной платформе, читается на другой.
- Формат не зависит от файловой системы.
- Порядок байтов и выравнивание фиксируются в формате хранилища.

### 5.5. Версионирование формата
- Формат хранилища имеет номер версии, записанный в каждом фблоке.
- Библиотека отказывается работать с хранилищем несовместимой мажорной версии.
- Изменения формата в рамках одной мажорной версии обратно совместимы на чтение.

## 6. Модель конкурентности

- К одному хранилищу в каждый момент времени обращается **ровно один экземпляр библиотеки**. Защита от одновременного открытия несколькими процессами реализуется через файловую блокировку или эксклюзивное открытие устройства.
- Внутри одного экземпляра: один логический писатель и несколько одновременных читателей; порядок между ними определяется приоритетом из 4.4.
- Один потребитель может работать с несколькими независимыми хранилищами одновременно, каждое — через свой экземпляр библиотеки.
- Восстановительная CLI-утилита (раздел 7) работает только с хранилищами, которые не открыты библиотекой.

## 7. Ключевые архитектурные решения

| Вопрос | Решение |
|--------|---------|
| Размер фблока | Фиксируется при инициализации |
| Размер фчанка (единица write-verify) | 4–16 МБ |
| Размер порции чтения | Существенно меньше фчанка; уточняется в документе операций |
| Порог битости фблока | Один битый фчанк = весь фблок битый |
| Идентификация | UUIDv4, генерируется до записи, хранится в индексах хранилища |
| Циклическая запись | Пропуск битых, защищённых и актуальных фблоков |
| Метрики | Сбор в памяти, доступ через API, без хранения истории |
| Восстановление | Отдельная CLI-программа, работает независимо |

## 8. Восстановление данных (требования к CLI-утилите)

- Чтение всего хранилища, включая повреждённые области.
- Извлечение фблоков по UUID.
- Возможность полного или частичного восстановления фблока.
- Работа в режиме read-only относительно исходного хранилища.

## 9. Расширение и сужение хранилища

### 9.1. Расширение
- Расширение выполняется добавлением новых фблоков в конец хранилища.
- Размер фблока не меняется.
- Существующие фблоки не перемещаются и не перезаписываются.
- Индексы новых фблоков становятся актуальны после подтверждения операции расширения.

### 9.2. Сужение

Сужение — **редкая и долгая** операция, выполняется потребителем осознанно. Размер фблока не меняется.

В зоне сужения (последние `M` фблоков хранилища) могут оказаться фблоки любого состояния, в том числе актуальные и защищённые. farc не отбрасывает такие данные молча — потребитель явно выбирает одну из стратегий:

| Стратегия | Поведение |
|-----------|-----------|
| **Перенос (copy)** | Актуальные и защищённые фблоки из зоны сужения копируются в свободные/неактуальные фблоки оставшейся зоны с сохранением UUID и состояния защиты. Если свободного места недостаточно — операция отменяется без изменений на диске. |
| **Отбрасывание (discard)** | Все фблоки зоны сужения теряются независимо от состояния. |

Перед запуском farc предоставляет потребителю сводку: сколько актуальных и защищённых фблоков попадает в зону сужения и сколько свободного места есть в оставшейся зоне. Потребитель решает, какую стратегию применить.

Операция должна быть возобновляемой после сбоя питания: промежуточное состояние сохраняется на диске, при следующем открытии хранилища farc обнаруживает незавершённое сужение и предлагает продолжить или откатить. Детали — в документе операций (`03-storage-operations.md`).

## 10. Импорт

- Импорт — копирование фблоков из одного хранилища в другое.
- Предусловие: размер фблока у источника и приёмника совпадает.
- Импортируются только целые, не битые фблоки.
- UUID импортированных фблоков сохраняются.
- При коллизии UUID (такой фблок уже есть в приёмнике) поведение задаётся параметром импорта: пропустить или заменить.
- Детали протокола импорта — в документе операций.
