# Журнал архитектурных решений (ADR)

## ADR-001: Расположение заголовка фблока

**Дата:** 2026-04-15
**Статус:** Принято

### Контекст
Необходимо хранить метаданные фблока (UUID, срок жизни). Рассматривались несколько вариантов: перед данными (внешний заголовок) и внутри области данных фблока.

### Варианты
1. **Перед фблоком (внешний заголовок):**
   - ✔ Точное соответствие заявленному размеру данных.
   - ❌ Усложняет расширение хранилища (размер заголовка в версии 2.0 может измениться, сломав расчет смещений).
   - ❌ Требует хранения таблицы смещений.

2. **Внутри фблока:**
   - ❌ Уменьшает полезную емкость фблока на размер заголовка.
   - ✔ Предельно простой расчет позиции: `Index * FixedSize`.  Поддерживает изменение размера хранилища.
   - ❌ при малом размере фблока, индексы могут занимать значительную его часть.

### Решение
Хранить заголовок внутри фблока.
**Обоснование:** Приоритет отдан надежности расчетов геометрии хранилища. Расширение или сужение хранилища предсказуемо, т.к. размер фблоков фиксировнный. Список только смещений для большого диска занимает слишком много места. 

## ADR-002: Стратегия хранения индексов хранилища

**Дата:** 2026-04-15
**Статус:** Принято

### Контекст
Требуется хранить карту состояния фблоков (битые/защищенные). Рассматривались варианты централизованного хранения в начале/конце файла и распределенного хранения.

### Анализ вариантов
1. **В конце хранилища:** Легко расширять, но медленный доступ (зона низких скоростей HDD).
2. **В начале хранилища:** Быстрый доступ, но невозможно расширить том без сдвига данных.
3. **Внешнее хранение (отдельный файл):** Риск рассинхронизации и потери восстанавливаемости.
4. **В начале каждого фблока (распределенные индексы):**
   - ✔ Равномерный износ диска (нет горячей зоны перезаписи индексов).
   - ✔ Данные о блоке физически рядом с блоком.

### Решение
Распределенное хранение индексов в заголовке каждого фблока.
**Обоснование:** Исключение бутылочного горлышка в одной физической зоне диска и атомарность (записали блок -> сразу обновили его индекс в том же IO).

## ADR-003: Выбор единицы чтения для HTTP API

**Дата:** 2026-04-15
**Статус:** Принято

### Контекст
При запросе короткого видеофрагмента (например, 10 секунд) система может либо вычитать с диска только нужные байты, либо вычитать весь фблок (например, 256 МБ) и отдать клиенту только часть.

### Сравнение
1. **Отдавать только часть (Range Read):**
   - ❌ StorageEngine должен знать внутренний формат контейнера (например, Matroska/MP4), чтобы корректно отдать самостоятельный кусок.
   - ❌ Сильная связность с форматом данных.
   - ✔ Экономия трафика и IO.

2. **Отдавать фблок целиком:**
   - ✔ StorageEngine остается тулзой общего назначения (Blob Store).
   - ✔ Потребитель (Application Layer) сам разбирает формат, что является его зоной ответственности.
   - ❌ Избыточное чтение с диска.

### Решение
Чтение на уровне StorageEngine выполняется **целыми фблоками** (или крупными фчанками).
**Обоснование:** Соблюдение принципа разделения ответственности (Single Responsibility). StorageEngine отвечает за надежность хранения и целостность фблока, а не за разбор мультимедийных контейнеров. Кроме того, это упрощает кэширование и снижает фрагментацию IO на HDD.

## ADR-004: Приоритет записи над чтением

**Дата:** 2026-04-15
**Статус:** Принято

### Решение
Запись имеет абсолютный приоритет. StorageEngine не может ставить запросы на чтение впереди ожидающей записи.
**Обоснование:** Предотвращение потери "живых" данных с камер из-за медленного диска, занятого раздачей архива. Система проектируется как "write intensive" с уклоном в надежность захвата.
