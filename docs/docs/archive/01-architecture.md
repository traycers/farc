# Архитектура системы хранения Farc

## 0. Обзор системы

**Farc** — система циклической записи видеопотоков с N источников на M независимых хранилищ.

- **Источники:** RTSP (опционально файлы для тестирования).
- **Выдача данных:** HTTP API (в перспективе RTSP).
- **Конфигурация:** Статический файл (JSON/YAML) на первое время.

### Режимы работы хранилища
1. **Циклическая запись** (перезапись по истечении срока актуальности).
2. **Запись до заполнения** (остановка записи с алертом при исчерпании места).

### Инициализация
Хранилище инициализируется однократно. На дисках 20 ТБ процесс занимает ~10+ минут.

### Модель данных: Фблок
Хранилище разбивается на последовательность **фблоков** фиксированного размера. Остаток в конце хранилища, меньший размера фблока, игнорируется. Фиксированный размер позволяет вычислять смещение любого фблока арифметически (`index * block_size`).

## 1. Структура хранилища и метаданные

### 1.1. Заголовок фблока
Заголовок, содержащий UUID, временные метки актуальности и флаги состояния, располагается **внутри фблока**. Это упрощает расчет смещений и поддержку операций расширения/сужения хранилища.

### 1.2. Индексы хранилища
Индексы (статусы фблоков: свободен/битый/защищен) хранятся исключительно **внутри хранилища** для гарантии восстанавливаемости без внешних зависимостей.

**Расположение индексов:** В начале каждого фблока (распределенный вариант).
Это обеспечивает равномерный износ диска (отсутствие "горячих" зон частой перезаписи в начале или конце тома) и консистентность данных.

## 2. Подсистема записи (Write Path)

### 2.1. Компоненты и поток данных
1. **WritePool:** Пул свободных буферов в памяти.
2. **Потребитель:** Заполняет буфер из пула данными.
3. **WriteQueue:** Очередь готовых к физической записи фблоков (FIFO).
4. **StorageEngine:** Монопольный владелец доступа к диску (либо пишет, либо читает).

### 2.2. Алгоритм физической записи
- **Выбор позиции:** Циклический поиск первого фблока со статусом: Не битый, Не защищенный, С истекшим сроком актуальности.
- **Процесс:**
    1. Запись осуществляется частями (**фчанками**, 4-16 МБ).
    2. **Write-Verify:** После записи каждого фчанка выполняется его чтение и побитовое сравнение.
    3. При несовпадении — повторная запись (до N попыток).
    4. При фатальном сбое записи фчанка — весь фблок помечается **битым**, запись переносится на следующий подходящий блок.

### 2.3. Обратное давление (Backpressure)
При заполнении WriteQueue потребитель получает сигнал перегрузки. Система не отбрасывает данные самостоятельно.

### 2.4. Уведомления
- **Перед началом перезаписи:** Событие об удалении (содержит UUID старого блока).
- **После успешной записи:** Событие о завершении (содержит UUID нового блока).

## 3. Подсистема чтения (Read Path)

### 3.1. Механика чтения
1. **ReadPool:** Пул буферов для чтения.
2. **Группировка запросов:** Запросы к одному фблоку группируются в очереди чтения для минимизации IOPS.
3. **Приоритет:** Операции записи прерывают операции чтения. Чтение выполняется в паузах записи.

### 3.2. API Чтения данных
Внешний API принимает запрос на диапазон данных (канал + время).
1. Потребитель по своей внешней БД определяет UUID целевого фблока.
2. Запрашивает у StorageEngine **весь фблок**.
3. Потребитель самостоятельно извлекает из фблока запрошенный диапазон (видео/аудио) и отправляет клиенту.

## 4. Консистентность и восстановление
- **Сбои питания:** Фблоки с незавершенной записью обнаруживаются фоновой проверкой и помечаются битыми.
- **Восстановление:** Вынесено в отдельную CLI-утилиту, способную сканировать сырое устройство и извлекать фблоки по известным сигнатурам.
