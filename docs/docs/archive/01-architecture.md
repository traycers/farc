# Архитектура системы хранения

## 0. Общая картина

Система (будем называть **farc**) пишет видео с N камер на M хранилищ. В каждое хранилище записывается видео с одинаковым параметром времени хранения. Получение видео ожидается по rtsp. Для тестов, хорошо бы поддержать работу с файлом (не обязательный параметр, облегчающий проверку системы). Предоставление фблоков ожидается по http api (в будущем возможно стоит добавить rtsp сервер).
Будем называть конфигурацией информацию необходиость писать с каких источников видео и в какие хранилища. Будет ли farc запрашивать конфигурацию или сам ходить за ней - не решено. Пока достаточно использовать json/yaml файл.

Система работает в двух режимах.
1. циклическая запись.
1. запись до заполнения хранилища. После заполнения хранилища система отправляет сообщение о заполнении хранилища.

Пользователь инициализирует хранилище начальными данными. После этого, хранилище готово к использованию. Ожидается, что на дисках 20ТБ это займёт 10+ минут, что приемлемо, ведь инициализация хранилища выполняется гораздо реже в сравнении с временем работы системы.

Хранилище разбивается на фблоки фиксированного размера. Если последний фблок будет меньше требуемого размера - тогда он игнорируется. Фиксированный размер нужен для расчёта смещения вместо его хранения, что позволит быстро получать позицию начала любого фблока.

### 0. Заголовок фблока

К фблоку нужен заголовок, описывающий фблок - период времени актуальности, id и тд. Заголовок можно расположить следующим образом.

#### перед фблоком
✔ - обеспечивает заявленный размер фблока.
❌ - при расширении сбивает расчёт положения фблока - требуется знать, до какого фблока шли смещения со старым размером, а после с увеличенным. Аналогично для сужения хранилища.
❌ - если хранить смещения фблоков, то они занимают много места в хранилище.
  
#### внутри фблока
❌ - уменьшает размер полезной нагрузки фблока - тех данных, которые может потребитель заполнить, ограничение необходимо передать потрбителю.
✔ - простота в расчётах смещений фблоков, включая расширение и сужение хранилища. 
✔ - возможность потребителю сообщить о слишком малом размере фблока.

Выбор пал на хранение заголовков внутри фблока из-за возможности простого расчёта смещений и изменения размера хранилища.

### 1. Индексы хранилища

Для хранилища потребуется хранить информацию, специфичную для него. Например, количество дней хранения, размер фблоков, битый ли фблок, защищён ли фблок от перезаписи. Такую информацию назовём `индексы хранилища`. 
Если их хранить только вне хранилища - тогда рискуем потерять индексы и сломать восстановление хранилища, но упрощаем их перезапись и ускоряем запуск сервиса, т.к. нет необходимости в их поиске. 
Хорошо, тогда будем хранить индексы и вне и в хранилище - тогда от нас потребуется внимательно следить за синхронизацией индексов и решать вопрос "какие из индексов актуальные - из хранилища или из стороннего места?". 
Хорошо, тогда будем хранить индексы только в хранилище - нет нужды в синхронизации индексов. Однако, где их располагать внутри хранилища? 

1. в конеце хранилища - простота в изменении размера хранилища, просто перезаписываем индексы в конец хранилища. Однако, конец хранилища - самая низкая скорость на HDD, а основное хранение ожидается как раз на HDD.
1. в начале - самый быстрый доступ на HDD, но за индексами располагаются фблоки, что значительно затрудняет расширение.
1. в начале, а дальше смещение на следующие индексы - позволит расширяться; при сужении не сможем преписать индексы в середину, т.к. там уже есть фблок; 
1. приватезировать первые индексы и в где-то в середине для индексов - предыдущие варианты ломали расчёт смещений, этот вариант сохраняет это; позволяет занять несколько фблоков подряд в начале и середине. Более простая перезапись индексов при изменении размера хранилища; требует определения в каких фблоках располагаются индексы.
1. в начале каждого фблока - предыдущие варианты требовали, после записи фблока, перехода в определённые места диска для записи индексов, а это увеличение числа операций записей в эти сектора диска по сравнению с остальными - не равномерный износ диска. В этом же варианте, такого недостатка нет. Однако, индексы могут занимать слишком большую часть фблока, что решается на этапе инициализации; много копий индексов - проблема в нахождении актуальной версии индексов; выходит некий список версий индексов.

### 2. Запись в хранилище

Выделяется два пула свободных фблоков - для записи (**WritePool**) и чтения (**ReadPool**). Потребитель забирает фблок из пула на запись, заполняет фблок, а после передаёт в очередь на запись (**WriteQueue**).
**StorageEngine** - некая сущность, эксклюзивно работающая с хранилищем (только она или читает или пишет), забирает фблок из очереди WriteQueue, добавляет необходимый заголовок и пишет в хранилище. Пишет порциями (**Fchunk**), после записи читает фчанк назад и сверяет прочитанное с тем, что в памяти. При совпадении продолжает запись. При отличии - пытается записать повторно в тоже место. После провала записи N раз, фблок считается битым, а StorageEngine переходит к перезаписи следующего фблока.

Потребитель заполняет WriteQueue готовыми фблоками, однако, если ОС выключится то эти фблоки будут утеряны. Нужен вариант частичной записи фблока с последующим дополнением оного.



## 1. Обзор компонентов

Система состоит из следующих логических компонентов:

**На стороне библиотеки:**

- **StorageEngine** - некая сущность, эксклюзивно работающая с хранилищем (только она или читает или пишет).
- **ReadPool** — пул свободных фблоков в памяти для чтения.
- **WritePool** — пул свободных фблоков в памяти для заполнения.
- **WriteQueue** — очередь заполненных фблоков в памяти, ожидающих записи на диск. Работает по принципу FIFO.
- **EventBus** — механизм уведомлений потребителя о событиях (новый фблок, удаление фблока, перегрузка очереди).
- **StorageDevice** — абстракция над физическим хранилищем (файл или блочное устройство). Предоставляет унифицированный интерфейс чтения/записи по смещению.

**Вне библиотеки:**

- **Потребитель** — внешний процесс, использующий API библиотеки.
- **Диск** — физическое хранилище (файл или раздел).

## 2. Потоки данных

В каждый момент времени StorageEngine или пишет в хранилище или читает из него. Приоритет на запись самый максимальный.

### 2.1. Поток записи

1. Потребитель получает фблок из WritePool и заполняет его.
1. Потребитель возвращает заполненный фблок и метаданные (время окончания актуальности) в WriteQueue.
1. StorageEngine выбирает целевой фблок для записи.
1. Если выбранный фблок защищён от перезаписи - в нём обновляются индексы на актуальные и выбирается следующий целевой фблок.
1. StorageEngine извлекает фблок из очереди.
1. Если выбранный фблок подлежит перезаписи — EventBus отправляет уведомление об удалении старого UUID.
1. Данные пишутся на диск пофчанково с проверкой (write-verify).
1. При обнаружении битого фчанка — весь фблок помечается битым, запись повторяется на другом фблоке.
1. После успешной записи:
    - EventBus отправляет уведомление об успешной записи фблока.
    - Буфер возвращается в WritePool.

  - `QUEUE_BACKPRESSURE` — очередь переполнена, началось обратное давление.
  - `QUEUE_NORMAL` — очередь вернулась к нормальному уровню.

## 3. Основные структуры в памяти

### 3.1. BufferPool

- Фиксированное количество буферов (задаётся в конфиге).
- Каждый буфер — непрерывная область памяти размером `fblock_size - header_size`.
- Состояния буфера: свободен, заполняется, готов к записи, в процессе записи.
- Счётчики для метрик: количество выданных буферов, время жизни буфера от выдачи до возврата.

### 3.2. WriteQueue

- Очередь указателей на заполненные буферы.
- Максимальный размер равен количеству буферов в пуле.
- Пороги для сигналов backpressure (например, 70% — предупреждение, 90% — перегрузка).

### 3.3. IndexManager

Хранит для каждого физического фблока (по индексу):
- UUID (нулевой — свободен).
- Состояние (свободен, занят, битый, защищён).
- Время окончания актуальности.
- Время записи.

Дополнительно:
- Маппинг UUID → индекс (хеш-таблица).
- Указатель последнего записанного индекса (для циклической записи).
- Счётчики битых фблоков.

## 4. Алгоритм выбора фблока для записи

1. Начать с индекса `last_written_index + 1`.
2. Для текущего индекса проверить:
   - Если фблок битый — перейти к следующему.
   - Если фблок защищён — перейти к следующему.
   - Если фблок свободен — выбрать его.
   - Если фблок занят:
     - Сравнить `текущее_время` с `время_окончания_актуальности`.
     - Если срок истёк — выбрать его (перед записью отправить уведомление об удалении).
     - Если срок не истёк — перейти к следующему.
3. Если обошли все фблоки и не нашли подходящего — авария «нет свободного места».
4. Обновить `last_written_index` на выбранный.

## 5. Обратное давление (backpressure)

### 5.1. Принцип

Библиотека не контролирует скорость поступления данных от потребителя. Если скорость наполнения превышает скорость записи на диск, очередь растёт. При достижении порогов библиотека сигнализирует потребителю о необходимости снизить темп.

### 5.2. Уровни загрузки

- **Норма** (0–70%): очередь в пределах нормы, работа без ограничений.
- **Предупреждение** (70–90%): очередь растёт, рекомендуется снизить темп наполнения.
- **Перегрузка** (>90%): критический уровень, потребитель должен применять стратегию деградации (отбрасывать менее важные данные, увеличивать паузы).

### 5.3. Механизм передачи сигнала

Сигнал передаётся синхронно при вызове `get_buffer()`:
- Возвращается буфер и флаг состояния.
- Потребитель сам решает, как реагировать.

Библиотека никогда не блокирует `get_buffer()` (если есть свободный буфер) и не отбрасывает данные самостоятельно.

## 6. Обработка битых фблоков

### 6.1. Обнаружение при записи

- Запись идёт пофчанково.
- После записи каждого фчанка — чтение и сравнение.
- При несовпадении:
  - Повтор записи (до N раз, по умолчанию 3).
  - Если все попытки исчерпаны — фчанк считается битым.
  - Весь фблок помечается битым в IndexManager.
  - Запись переносится на следующий подходящий фблок.

### 6.2. Мониторинг

- IndexManager хранит количество битых фблоков.
- При превышении порога (по умолчанию 5%) генерируется алерт через EventBus.
- Работа продолжается, битые фблоки пропускаются при циклической записи.

## 7. Восстановление после сбоя

### 7.1. Проблема незавершённой записи

При сбое питания в момент записи фблок может оказаться в промежуточном состоянии: часть фчанков записана, часть — нет. Такой фблок непригоден для использования.

### 7.2. Механизм обнаружения

- В заголовке фблока есть флаг «в процессе записи».
- Перед началом записи флаг устанавливается.
- После успешной записи всех фчанков флаг снимается и устанавливается флаг «записан».

### 7.3. Действия при старте

1. IndexManager загружает последнюю консистентную копию индексов.
2. Сканируются заголовки всех фблоков (может выполняться в фоне).
3. Фблоки с флагом «в процессе» помечаются как битые в IndexManager.
4. Обновляются копии индексов на диске.

## 8. Конкурентность и потоки

### 8.1. Поток записи

- Один выделенный поток, который извлекает буферы из WriteQueue и пишет их на диск.
- Имеет абсолютный приоритет над чтением.

### 8.2. Поток чтения

- Может выполняться в том же потоке, что и вызов API, или в отдельном пуле.
- Чтение выполняется только когда нет активной записи (поток записи спит в ожидании очереди).

### 8.3. Поток уведомлений

- Отдельный поток для асинхронной отправки событий потребителю.
- Не блокирует основные операции записи/чтения.

### 8.4. Синхронизация

- IndexManager защищён мьютексом (или RWLock), так как читается и пишется из разных потоков.
- WriteQueue — lock-free очередь или защищена мьютексом.
- BufferPool — мьютекс для операций выделения/возврата.

## 9. Метрики

### 9.1. Сбор

Все метрики хранятся в памяти в виде атомарных счётчиков или кольцевых буферов (для скользящих средних).

### 9.2. Состав

**Метрики записи:**
- Количество записанных фблоков.
- Время записи фблока (последнее, среднее за период).
- Количество повторов записи фчанков.
- Скорость записи (байт/сек).

**Метрики очереди:**
- Текущий размер очереди.
- Пиковый размер очереди.
- Время ожидания буфера в очереди.
- Количество переходов между уровнями backpressure.

**Метрики состояния:**
- Всего фблоков.
- Количество и процент битых фблоков.
- Количество защищённых фблоков.

### 9.3. Доступ

- Метрики доступны через API библиотеки.
- Потребитель сам решает, как их экспортировать (логи, Prometheus, файл).

## 10. Точки расширения

### 10.1. Абстракция StorageDevice

Позволяет добавить поддержку новых типов хранилищ:
- Файл (Windows/Linux).
- Блочное устройство (Linux).
- Сетевое блочное устройство (iSCSI, NBD).
- Разреженный файл.

### 10.2. Стратегия выбора фблока

Текущая — циклическая с учётом актуальности. Возможные расширения:
- Приоритет по близости к головке (для HDD).
- Группировка по времени актуальности.

### 10.3. Стратегия backpressure

Текущая — сигнализация потребителю. Возможное расширение:
- Блокировка `get_buffer()` при критическом переполнении.
