# Алгоритмы работы хранилища

## 1. Инициализация хранилища

### 1.1. Создание нового хранилища

**Входные параметры:**
- Путь к файлу или блочному устройству.
- Размер фблока (по умолчанию 2 ГБ).
- Размер фчанка (по умолчанию 16 МБ).
- Количество копий индексов (по умолчанию 4).
- Признак принудительной инициализации (force).

**Алгоритм:**

1. Проверить, что устройство/файл доступно для записи.
2. Если цель — файл и он не существует:
   - Создать файл.
   - Выделить место заданного размера (fallocate / SetFileValidData).
3. Если цель — блочное устройство:
   - Определить его размер.
   - Проверить, что размер достаточен для размещения хотя бы одного фблока и метаданных.
4. Проверить, не содержит ли устройство/файл уже сигнатуру хранилища:
   - Прочитать первые 8 байт.
   - Если сигнатура "FBLOCK01" обнаружена и флаг force не установлен — ошибка.
5. Рассчитать общее количество фблоков:
   - `total_fblocks = floor(размер_устройства / fblock_size)`.
   - Убедиться, что `total_fblocks >= 2 * index_copies + 1` (минимум один фблок для данных).
6. Создать суперблок в памяти:
   - Заполнить все поля согласно параметрам.
   - Установить флаг dirty shutdown (0x02).
   - Вычислить и записать контрольную сумму.
7. Записать суперблок на диск (смещение 0).
8. Создать начальный набор индексов в памяти:
   - Массив из `total_fblocks` записей.
   - Все записи инициализируются нулями (свободные фблоки).
9. Для каждой копии (i от 1 до index_copies):
   - Создать фблок типа "индексы" с номером копии i.
   - Записать в него текущий набор индексов.
   - Поместить в начало (индекс i-1) или в конец (индекс total_fblocks - index_copies + i - 1).
   - Обновить поле `index_fblocks` в суперблоке.
10. Записать вторую группу копий в конец хранилища.
11. Установить `active_index_fblock` в суперблоке на первую копию.
12. Записать обновлённый суперблок на диск.
13. Выполнить синхронизацию (fsync / FlushFileBuffers).

**Результат:**
- Хранилище готово к работе.
- Суперблок записан, копии индексов созданы.
- Все фблоки данных помечены как свободные.

### 1.2. Открытие существующего хранилища

**Алгоритм:**

1. Открыть устройство/файл.
2. Прочитать суперблок (первые 4096 байт).
3. Проверить сигнатуру "FBLOCK01". Если отсутствует — ошибка "не хранилище".
4. Проверить контрольную сумму суперблока. При несовпадении — ошибка "повреждённый суперблок".
5. Проверить версию формата. Если версия не поддерживается — ошибка.
6. Загрузить параметры из суперблока (размеры, количество фблоков).
7. Найти активную копию индексов:
   - Прочитать все фблоки из списка `index_fblocks`.
   - Для каждой копии проверить целостность.
   - Выбрать копию с максимальным `copy_time`.
8. Загрузить индексы выбранной копии в память.
9. Если в суперблоке установлен флаг dirty shutdown (0x02):
   - Запустить фоновое сканирование незавершённых фблоков (раздел 4).
10. Обновить `active_index_fblock` в суперблоке, если он изменился.
11. Установить флаг dirty shutdown (0x02) в памяти, записать на диск (означает, что система работает).

**Результат:**
- Индексы загружены в память.
- Хранилище готово к операциям чтения и записи.

## 2. Запись фблока

### 2.1. Основной алгоритм

**Входные параметры:**
- Указатель на буфер с данными.
- Время окончания актуальности (expiry_time).

**Алгоритм:**

1. Заблокировать IndexManager для выбора целевого фблока.
2. Вызвать алгоритм выбора следующего фблока (раздел 2.2).
3. Если выбор вернул "нет места" — разблокировать IndexManager и вернуть ошибку.
4. Получить индекс выбранного фблока.
5. Проверить, был ли фблок занят:
   - Если да, отправить уведомление об удалении через EventBus (с UUID старого фблока).
6. В IndexManager пометить фблок как "в процессе записи" (флаг 0x02).
7. Разблокировать IndexManager.
8. Записать заголовок фблока с флагом "в процессе" на диск.
9. Для каждого фчанка в фблоке:
   - Вызвать алгоритм записи фчанка с проверкой (раздел 2.3).
   - Если фчанк записать не удалось:
     - Прервать цикл.
     - Вызвать алгоритм обработки битого фблока (раздел 2.4).
     - Начать запись заново с шага 1.
10. Все фчанки записаны успешно:
    - Сгенерировать UUID (UUIDv4).
    - Заполнить заголовок фблока: UUID, expiry_time, write_time, data_size.
    - Снять флаг "в процессе", установить флаг "записан".
    - Записать финальный заголовок на диск.
11. Заблокировать IndexManager.
12. Обновить запись индекса для этого фблока:
    - UUID, expiry_time, write_time, флаги (записан).
13. Обновить указатель `last_written_index` на текущий индекс.
14. Разблокировать IndexManager.
15. Вызвать алгоритм синхронизации индексов (раздел 2.5).
16. Отправить уведомление о новом фблоке через EventBus.
17. Вернуть UUID потребителю.

### 2.2. Выбор следующего фблока

**Алгоритм:**

1. Начать с индекса `(last_written_index + 1) % total_fblocks`.
2. Установить счётчик просмотренных фблоков = 0.
3. Пока счётчик < total_fblocks:
   - Получить запись индекса для текущего индекса.
   - Если фблок битый (флаг 0x04) — перейти к следующему.
   - Если фблок защищён (флаг 0x08) — перейти к следующему.
   - Если фблок свободен (UUID нулевой) — вернуть этот индекс.
   - Если фблок занят (флаг 0x01):
     - Получить `expiry_time` из записи.
     - Если `текущее_время >= expiry_time` — вернуть этот индекс.
     - Иначе — перейти к следующему.
   - Увеличить счётчик.
   - Перейти к следующему индексу (циклически).
4. Если цикл завершился без результата — вернуть "нет места".

### 2.3. Запись фчанка с проверкой

**Входные параметры:**
- Смещение фчанка в хранилище.
- Указатель на данные фчанка.
- Размер фчанка.

**Константы:**
- `max_retries = 3` (настраивается).

**Алгоритм:**

1. Установить счётчик попыток = 0.
2. Пока счётчик < max_retries:
   - Записать данные фчанка на диск по указанному смещению.
   - Выполнить синхронизацию (fsync / FlushFileBuffers) для этого диапазона.
   - Прочитать данные обратно с диска во временный буфер.
   - Сравнить прочитанное с оригиналом (memcmp).
   - Если данные совпадают — вернуть успех.
   - Увеличить счётчик попыток.
3. Если все попытки исчерпаны — вернуть ошибку "битый фчанк".

### 2.4. Обработка битого фблока

**Входные параметры:**
- Индекс битого фблока.

**Алгоритм:**

1. Заблокировать IndexManager.
2. Пометить фблок как битый в индексах:
   - Снять флаг "в процессе" (0x02).
   - Установить флаг "битый" (0x04).
3. Увеличить счётчик битых фблоков.
4. Разблокировать IndexManager.
5. Вызвать синхронизацию индексов (раздел 2.5).
6. Отправить алерт о новом битом фблоке через EventBus.
7. Проверить порог битых фблоков:
   - Если `(битые_фблоки / total_fblocks) * 100 >= порог (5%)` — отправить критический алерт.

**Примечание:** после обработки битого фблока запись текущего буфера начинается заново с выбора следующего фблока.

### 2.5. Синхронизация индексов на диск

**Алгоритм:**

1. Заблокировать IndexManager (если ещё не заблокирован).
2. Сформировать полный образ индексов в памяти:
   - Массив из `total_fblocks` записей.
   - Сериализовать текущее состояние IndexManager.
3. Установить `copy_time = текущее_время`.
4. Для каждой копии из списка `index_fblocks` в суперблоке:
   - Сформировать данные копии (copy_time, copy_number, entry_count, checksum, entries).
   - Записать данные в соответствующий фблок (как обычную запись фблока типа "индексы", но без проверки фчанков — индексы маленькие).
   - Обновить `write_time` в заголовке фблока с индексами.
5. Обновить `active_index_fblock` в суперблоке (на последнюю записанную копию).
6. Обновить `last_update` в суперблоке.
7. Записать суперблок на диск.
8. Разблокировать IndexManager.

**Оптимизация:** синхронизация индексов может выполняться не после каждой записи фблока, а с накоплением (например, раз в 10 записей или раз в 5 секунд). Это снижает накладные расходы. При сбое теряется информация о последних записанных фблоках, но они останутся на диске и могут быть восстановлены сканированием.

## 3. Чтение фблока

### 3.1. Полное чтение

**Входные параметры:**
- UUID фблока.

**Алгоритм:**

1. Заблокировать IndexManager для чтения.
2. Найти индекс фблока по UUID в маппинге.
3. Если UUID не найден — разблокировать IndexManager и вернуть ошибку.
4. Получить запись индекса.
5. Проверить, что фблок не битый (флаг 0x04).
6. Получить `data_size` из записи.
7. Разблокировать IndexManager.
8. Выделить буфер размером `data_size`.
9. Рассчитать физическое смещение: `индекс * fblock_size + 128` (пропуск заголовка).
10. Прочитать `data_size` байт с диска по смещению.
11. Проверить контрольную сумму данных (xxHash64) — сравнить с хранимой в заголовке.
12. Если сумма не совпадает — вернуть ошибку "повреждённые данные".
13. Вернуть буфер с данными.

### 3.2. Частичное чтение

**Входные параметры:**
- UUID фблока.
- Смещение внутри данных (offset).
- Размер для чтения (size).

**Алгоритм:**

1. Заблокировать IndexManager для чтения.
2. Найти индекс фблока по UUID.
3. Проверить, что фблок не битый.
4. Получить `data_size`.
5. Разблокировать IndexManager.
6. Проверить, что `offset + size <= data_size`.
7. Выделить буфер размером `size`.
8. Рассчитать физическое смещение: `индекс * fblock_size + 128 + offset`.
9. Прочитать `size` байт с диска.
10. Вернуть буфер.

**Примечание:** при частичном чтении контрольная сумма данных не проверяется, так как для этого нужно читать все данные. Проверка целостности заголовка (CRC32C) всё равно выполняется при открытии фблока.

## 4. Восстановление после сбоя

### 4.1. Обнаружение незавершённых фблоков

**Контекст:** вызывается при старте системы, если суперблок содержит флаг dirty shutdown.

**Алгоритм:**

1. Для каждого индекса от 0 до total_fblocks-1:
   - Прочитать заголовок фблока (128 байт).
   - Проверить магическое число заголовка (0xFB1001).
   - Если магическое число не совпадает — пропустить (свободный фблок).
   - Проверить флаги:
     - Если установлен флаг "в процессе" (0x02) — добавить индекс в список незавершённых.
2. Для каждого индекса в списке незавершённых:
   - Заблокировать IndexManager.
   - Пометить фблок как битый (флаг 0x04).
   - Снять флаг "в процессе" (0x02).
   - Очистить UUID.
   - Увеличить счётчик битых фблоков.
   - Разблокировать IndexManager.
3. Вызвать синхронизацию индексов.
4. Записать в лог количество найденных и помеченных незавершённых фблоков.

**Оптимизация:** сканирование может выполняться в фоновом потоке, не блокируя запуск основных операций. В этом случае система начинает работу сразу после загрузки индексов, а помеченные как битые фблоки обновляются позже.

### 4.2. Восстановление индексов из заголовков фблоков

**Контекст:** если все копии индексов повреждены или отсутствуют.

**Алгоритм:**

1. Создать новый пустой набор индексов в памяти (все записи нулевые).
2. Для каждого индекса от 0 до total_fblocks-1:
   - Прочитать заголовок фблока.
   - Проверить магическое число и контрольную сумму заголовка.
   - Если заголовок валиден и установлен флаг "записан" (0x01):
     - Скопировать UUID, expiry_time, write_time, флаги в запись индекса.
     - Если установлен флаг "битый" (0x04) — пометить как битый.
     - Если установлен флаг "защищён" (0x08) — пометить как защищённый.
3. После сканирования всех фблоков:
   - Вызвать синхронизацию индексов (записать все копии заново).
   - Обновить суперблок (снять dirty shutdown).

**Примечание:** этот алгоритм используется утилитой восстановления, а не основной библиотекой при обычном запуске.

## 5. Расширение хранилища

### 5.1. Алгоритм расширения

**Входные параметры:**
- Новый размер хранилища (должен быть больше текущего).

**Алгоритм:**

1. Проверить, что новый размер кратен `fblock_size`.
2. Увеличить размер файла/устройства (fallocate / SetFileValidData).
3. Рассчитать новое количество фблоков: `new_total = новый_размер / fblock_size`.
4. Заблокировать IndexManager.
5. Увеличить размер массива индексов в памяти до `new_total`.
6. Инициализировать новые записи нулями (свободные фблоки).
7. Разблокировать IndexManager.
8. Создать новые копии индексов в конце хранилища:
   - Для каждой копии (i от 1 до index_copies):
     - Индекс фблока = `new_total - index_copies + i - 1`.
     - Записать текущий набор индексов в этот фблок (тип "индексы").
     - Добавить индекс в список `index_fblocks` в суперблоке.
9. Обновить поля в суперблоке:
   - `total_fblocks = new_total`.
   - `last_update = текущее_время`.
10. Пересчитать контрольную сумму суперблока.
11. Записать суперблок на диск.

**Примечание:** расширение выполняется без остановки записи. Единственная блокировка — на время увеличения массива индексов в памяти (миллисекунды).

## 6. Завершение работы

### 6.1. Чистое завершение

**Алгоритм:**

1. Прекратить приём новых буферов от потребителя.
2. Дождаться опустошения WriteQueue (записать все ожидающие буферы).
3. Вызвать синхронизацию индексов (записать все копии).
4. Обновить суперблок:
   - Установить флаг clean shutdown (0x01).
   - Снять флаг dirty shutdown (0x02).
   - Обновить `last_update`.
   - Пересчитать контрольную сумму.
5. Записать суперблок на диск.
6. Выполнить fsync / FlushFileBuffers для всего устройства/файла.
7. Освободить ресурсы и закрыть устройство/файл.
