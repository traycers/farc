# Операции хранилища

## 1. Назначение документа

Документ описывает алгоритмы операций над Storage: инициализацию, поиск актуальных индексов при старте, ConsistencyCheck, выбор фблока для записи, запись, чтение, расширение, сужение, импорт.

Документ опирается на:
- Концептуальную модель из `01-storage.md`.
- Бинарный формат из `02-storage-format.md`.
- ADR-006 (ленивая инициализация), ADR-007 (внешний каталог), ADR-008 (write_sequence).

Документ **не описывает** публичный API и протоколы взаимодействия с потребителем.

## 2. Конфигурация

### 2.1. Конфигурация farc (один сервис на сервер)

Конфигурация сервиса farc хранится в отдельном файле и содержит:

- Список Storage:
  - идентификатор Storage;
  - путь к файлу или блочному устройству;
  - путь к файлу каталога на SSD (опционально, см. ADR-007);
- Параметры HTTP-сервера, метрик, событий.

Пути к каталогам на SSD задаются здесь, а не в самом Storage. Это позволяет переносить Storage между серверами без необходимости править внутренние данные хранилища.

### 2.2. Параметры Storage (в фблоке 0)

Параметры, влияющие на работу с конкретным Storage, хранятся в JSON в прологе фблока 0 (см. `02-storage-format.md`, раздел 5.2).

## 3. Инициализация Storage

Инициализация выполняется однократно при создании Storage. Реализуется лениво (ADR-006).

### 3.1. Алгоритм

1. **Проверка пустоты.** Прочитать первые 8 байт целевого файла/устройства. Если там валидный `magic_prolog` — возврат ошибки «уже инициализирован». Принудительная инициализация требует явного флага `--force`.

2. **Открытие хранилища:**
   - Файловый режим: создать файл нужного размера, выполнить `fallocate` (Linux) или эквивалент для резервирования места.
   - Блочное устройство: открыть для записи, проверить размер.

3. **Расчёт геометрии:**
   - `N = storage_size / fblock_size` — количество фблоков.
   - `catalog_size = N × 33` (см. формат каталога).

4. **Подготовка фблока 0 в памяти:**
   - Фиксированная часть пролога: `magic_prolog`, версия v1.0, `write_sequence = 1`, `catalog_time = now()`, `fblock_size`, `params_size`, `catalog_entry_count = N`, `catalog_size`.
   - Параметры в JSON.
   - Каталог: для индекса 0 — `flags = in_progress`, для всех остальных — нули (`uninitialized`).
   - CRC32 для трёх частей заголовка.
   - Эпилог с `magic_epilog`. Размер TOC и данных — нулевой.

5. **Запись фблока 0:**
   - Запись пофчанково с верификацией.
   - Если запись провалилась — инициализация отменяется, ошибка.

6. **Создание каталога на SSD** (если путь указан в конфиге farc):
   - Создать файл размером `N × 33` байт.
   - Заполнить нулями (все фблоки кроме нулевого — `uninitialized`).
   - Записать запись индекса 0 как `ready` с UUID и временами фблока 0.
   - Подсчитать и записать контрольную сумму каталога (см. `02-storage-format.md`, описание формата каталога SSD).
   - `fsync`.

7. Storage готов к работе. Все фблоки кроме нулевого находятся в состоянии `uninitialized`.

### 3.2. Свойства

- Время инициализации: секунды независимо от размера Storage.
- На основном диске пишется только фблок 0.
- Каталог на SSD создаётся целиком, но это десятки МБ — секунды на SSD.

## 4. Старт Storage

При каждом открытии Storage необходимо восстановить актуальные индексы в памяти `IndexManager`. Алгоритм имеет основной путь и два fallback'а с убывающей скоростью.

### 4.1. Путь 1: загрузка из каталога SSD (основной)

1. Открыть файл каталога по пути из конфигурации farc.
2. Прочитать каталог целиком.
3. Проверить контрольную сумму каталога.
4. Если сумма валидна — загрузить индексы в `IndexManager`. Перейти к ConsistencyCheck (раздел 5).
5. Если сумма невалидна или файл отсутствует — путь 2.

### 4.2. Путь 2: сканирование заголовков основного хранилища (fallback)

Используется, когда каталог SSD недоступен или повреждён. Состоит из двух фаз: лёгкое сканирование для определения свежайшего фблока и полное чтение его каталога.

#### 4.2.1. Фаза 1: определение размера фблока

Размер фблока неизвестен заранее (он в параметрах фблока 0).

1. Прочитать 56 байт со смещения 0. Если `magic_prolog` валиден — это фблок 0, прочитать его параметры, взять `fblock_size`.
2. Если фблок 0 повреждён — последовательно читать первые 56 байт со смещений `K × MIN_FBLOCK_SIZE` (где `MIN_FBLOCK_SIZE` — минимальный допустимый размер фблока, фиксированный константой формата) и искать первый валидный пролог. По параметрам этого фблока определить `fblock_size`.

Если ни один валидный пролог не найден — Storage считается полностью повреждённым, требуется восстановление через CLI-утилиту.

#### 4.2.2. Фаза 2: поиск свежайшего фблока

1. Для каждого индекса `K` от 0 до `N-1` прочитать «лёгкую часть» пролога: 32 байта со смещения `K × fblock_size`, содержащие `magic_prolog`, `format_version`, `write_sequence` и `catalog_time`.
2. Отобрать фблоки с валидным `magic_prolog`.
3. Среди них выбрать фблок с максимальным `write_sequence` (ADR-008).

#### 4.2.3. Фаза 3: чтение каталога свежайшего фблока

1. Прочитать заголовок выбранного фблока целиком (фиксированная часть + параметры + каталог + контрольные суммы заголовка).
2. Проверить три CRC32:
   - Если все три валидны — индексы загружены, ConsistencyCheck.
   - Если CRC каталога невалиден — выбрать следующий по `write_sequence` фблок, повторить с шага 1.
   - Если все фблоки исчерпаны — Storage полностью повреждён.

#### 4.2.4. Реализация

Чтение прологов в фазе 2 — основная нагрузка fallback-пути.

- **Linux:** io_uring для параллельной подачи запросов на чтение прологов всех фблоков. Параллельность снижает влияние seek-задержек HDD.
- **Windows и другие платформы:** последовательное чтение через стандартную библиотеку. Для миллиона фблоков на HDD это десятки минут.

Производительность fallback'а — компромиссная. Основной путь работы — каталог SSD.

### 4.3. Восстановление каталога SSD

Если Storage стартовал по пути 2 (каталог отсутствовал или был повреждён), после успешной загрузки индексов и ConsistencyCheck каталог SSD пересоздаётся:

1. Создать файл каталога заново (или перезаписать).
2. Записать актуальное состояние из `IndexManager`.
3. `fsync`.

После этого следующий старт пройдёт по пути 1.

## 5. ConsistencyCheck

ConsistencyCheck выполняется один раз при старте Storage, после загрузки индексов любым из путей. Цель — обнаружить и обработать фблоки в состоянии `in_progress` (потенциально незавершённые после сбоя питания).

### 5.1. Алгоритм

1. **Определить кандидата.** В индексах находится фблок с состоянием `in_progress`. Это всегда последний записанный фблок: согласно ADR-002, его подтверждение в `ready` должно было прийти в каталоге следующего фблока, но следующего нет.

2. **Проверка целостности фблока.** Прочитать заголовок и эпилог фблока:
   - Если `magic_epilog` отсутствует — запись не дошла до эпилога. Перевести фблок в `bad`.
   - Если `magic_epilog` валиден, проверить CRC32 данных и TOC:
     - `crc32_data` валиден и `crc32_toc` валиден — данные целы. Перевести фблок в `ready` в индексах.
     - Иначе — данные повреждены. Перевести фблок в `bad`.

3. **Зафиксировать решение.** Изменения сохраняются в `IndexManager` в памяти. На диск изменения попадут при следующей записи фблока (ADR-002). Каталог SSD обновляется немедленно.

### 5.2. Случай нескольких `in_progress`

В норме `in_progress` фблок один. Если их обнаружено несколько (это возможно только при глубоком повреждении хранилища) — все, кроме фблока с максимальным `write_sequence`, помечаются `bad` без проверки.

### 5.3. Свойства

- Не выполняет никаких записей на основной диск.
- Длительность: миллисекунды (один фблок).
- Никогда не блокирует переход Storage в running надолго.

## 6. Выбор фблока для записи

Алгоритм выполняется компонентом `IndexManager` по запросу `Recorder`'а перед каждой записью.

### 6.1. Состояние

`IndexManager` поддерживает:
- Полную карту индексов (состояния всех фблоков).
- `cursor` — последний физический индекс, на который писалось. Восстанавливается при старте: индекс фблока с максимальным `write_sequence`.

### 6.2. Алгоритм

```
function select_next_index(indexes, cursor, retention_days, now):
    n = len(indexes)
    
    # Приоритет 1: первый uninitialized по кругу от cursor+1
    for i in range(n):
        idx = (cursor + 1 + i) mod n
        if indexes[idx].state == uninitialized:
            return idx
    
    # Приоритет 2: первый ready с истёкшим сроком хранения, не protected
    for i in range(n):
        idx = (cursor + 1 + i) mod n
        e = indexes[idx]
        if e.state == ready and not e.protected and (now - e.end >= retention_days):
            return idx
    
    return NO_SPACE
```

### 6.3. Гарантии

- Циклическое движение от позиции последней записи.
- Приоритет `uninitialized` — первый полный проход заполняет диск без перезаписей.
- Только после заполнения всех `uninitialized` начинается циклическая перезапись `ready` фблоков по сроку хранения.

### 6.4. Алерт «нет места»

Если `select_next_index` вернул `NO_SPACE`, `Recorder` публикует событие `storage.alert` с `severity=critical` и `reason=no_free_fblocks`. Storage продолжает работать (буфер записи остаётся в очереди), новый вызов будет повторён при изменении состояния (например, истёк срок хранения какого-то фблока).

## 7. Запись фблока

Расширение последовательности из 6.1 в `01-storage.md` с конкретикой по операциям диска.

### 7.1. Подготовка

`Recorder` уже получил буфер от потребителя с заполненными данными и метаданными `begin`/`end`. Дальнейшие шаги:

1. Запросить у `IndexManager` индекс N (раздел 6).
2. Перевести `IndexManager.indexes[N]` в `in_progress`.
3. Опубликовать `fblock.write.started`.
4. Если `indexes[N]` ранее был `ready` с непустым UUID — опубликовать `fblock.deleted` со старым UUID.

### 7.2. Сборка фблока в памяти

Recorder формирует полный образ фблока в буфере:

1. Фиксированная часть пролога: `magic_prolog`, версия, `write_sequence = max_seen + 1`, `catalog_time = now()`, остальные поля.
2. Параметры (JSON).
3. Каталог: текущий снимок индексов из `IndexManager` (где сам N — `in_progress`).
4. Три CRC32 заголовка.
5. Данные потребителя (payload).
6. TOC (формат непрозрачен для Storage).
7. Эпилог с `magic_epilog` и CRC32 данных и TOC.

### 7.3. Запись на диск

Запись выполняется пофчанково с верификацией (write-verify):

1. Для каждого фчанка:
   - Записать фчанк через `IoBackend`.
   - Прочитать фчанк обратно через `IoBackend`.
   - Сравнить с исходным буфером.
   - При несовпадении — фчанк битый. Перевести фблок N в `bad`, опубликовать `fblock.write.failed` (внутреннее событие), вернуться к шагу 7.1.1 для выбора нового индекса. Буфер из очереди не выгружается.

2. Если все фчанки записаны успешно:
   - В `IndexManager` зафиксировать `indexes[N] = ready` с UUID и временами.
   - Обновить запись `N` в каталоге SSD, `fsync`.
   - Опубликовать `fblock.write.completed`.
   - Вернуть буфер в пул.

Подтверждение `in_progress → ready` на **основном диске** появится только при записи следующего фблока (ADR-002). Промежуточный сбой питания между шагами 1 и 2 оставит фблок в состоянии `in_progress` — ConsistencyCheck при следующем старте проверит его эпилог и завершит переход.

### 7.4. Размер фчанка и порция чтения

- Размер фчанка фиксируется параметром `fchunk_size` в JSON параметров.
- Размер порции чтения — параметром `read_chunk_size`. По умолчанию равен `fchunk_size`.
- Оба должны быть кратны размеру кластера диска (типично 4 КБ).

## 8. Чтение фблока

Многоэтапное чтение (ADR-004): метаданные → выбор данных потребителем → чтение данных порциями.

### 8.1. Этап 1: метаданные

1. Потребитель запрашивает чтение по UUID.
2. `Reader` запрашивает у `IndexManager` физический индекс. `IndexManager` возвращает индекс только для фблоков в состоянии `ready`.
3. Если состояние не `ready` (uninitialized/in_progress/bad) — ошибка чтения.

### 8.2. Этап 2: чтение TOC

Если потребитель запросил TOC (для определения, какие диапазоны он будет читать):

1. `Reader` через `StorageEngine` читает область TOC фблока (смещение от конца фблока, см. `02-storage-format.md`).
2. Чтение разбивается на порции размером `read_chunk_size`. Между порциями `StorageEngine` проверяет очередь записи и при необходимости выполняет запись.

### 8.3. Этап 3: чтение диапазонов данных

1. Потребитель указывает список диапазонов `[(offset, length), ...]`.
2. `Reader` группирует и сортирует диапазоны для оптимизации seek'ов.
3. `StorageEngine` читает каждый диапазон порциями `read_chunk_size`.
4. Между порциями: проверка очереди записи, при необходимости — запись фблока целиком, затем продолжение чтения.

### 8.4. Гарантии

- Запись никогда не ждёт чтение дольше одной порции (`read_chunk_size`).
- Чтение никогда не прерывает уже начатую запись фчанка.

## 9. Расширение Storage

Расширение увеличивает количество фблоков в Storage. Размер фблока не меняется. Выполняется над остановленным Storage.

### 9.1. Алгоритм

1. **Остановить Storage.** Archive переводит Storage в `stopped`.
2. **Расширить файл/раздел:**
   - Файл: `truncate` до нового размера + `fallocate` нового пространства.
   - Блочное устройство: проверить новый размер устройства (расширение делается администратором на уровне ОС).
3. **Расчёт нового количества фблоков:** `N_new = new_storage_size / fblock_size`.
4. **Расширить каталог SSD:** дописать нулевые байты до размера `N_new × 33`. Все новые записи каталога автоматически становятся `uninitialized` (нулевой байт flags).
5. **Обновить контрольную сумму каталога SSD.**
6. **Запустить Storage.** В первом записанном фблоке после расширения новое значение `catalog_entry_count = N_new` появится на основном диске.

### 9.2. Свойства

- На основном диске никаких записей не выполняется (только увеличение размера файла).
- Время операции — секунды (truncate + расширение каталога SSD).
- Storage недоступен только на время остановки и расширения.

## 10. Сужение Storage

Сужение уменьшает количество фблоков. Долгая операция, выполняется над остановленным Storage. Стратегия выбирается потребителем (см. 9.2 requirements).

### 10.1. Подготовка

1. **Остановить Storage.**
2. **Подсчитать содержимое зоны сужения:** последние `M` фблоков, где `M = N - N_new`.
3. **Сводка для потребителя:**
   - Количество `ready` (с разбивкой на актуальные по сроку хранения и истёкшие).
   - Количество `protected`.
   - Количество `uninitialized` и `bad`.
   - Количество доступных для записи фблоков в оставшейся зоне (для оценки возможности переноса).
4. **Потребитель выбирает стратегию:** `copy` (перенос актуальных и `protected`) или `discard` (отбросить всё).

### 10.2. Стратегия `discard`

1. Truncate файла основного хранилища до `N_new × fblock_size`.
2. Truncate файла каталога SSD до `N_new × 33` байт.
3. Обновить контрольную сумму каталога.
4. Запустить Storage.

### 10.3. Стратегия `copy`

Долгая операция, требующая возобновляемости.

#### 10.3.1. Журнал сужения

Журнал — отдельный файл рядом с каталогом SSD. Содержит:
- Версия операции.
- `N_old`, `N_new`.
- Список «миссий» (mission): пар `(source_index, target_index)` для копирования.
- Статус каждой миссии: `pending`, `in_progress`, `done`, `failed`.

Журнал записывается с `fsync` после каждого изменения статуса миссии.

#### 10.3.2. Планирование миссий

1. Найти в зоне сужения все фблоки в состоянии `ready` (актуальные и `protected`).
2. Найти в оставшейся зоне доступные для записи фблоки.
3. Если доступного места недостаточно — вернуть ошибку, операция отменена.
4. Создать миссии: для каждого фблока зоны сужения — целевой индекс в оставшейся зоне.
5. Записать журнал.

#### 10.3.3. Выполнение миссий

Для каждой миссии в журнале со статусом `pending`:

1. Перевести статус в `in_progress`, `fsync` журнала.
2. Прочитать фблок-источник из основного хранилища.
3. Записать его на целевой индекс с тем же UUID, временами, флагом `protected`.
4. Перезаписать соответствующую запись в каталоге SSD.
5. Перевести статус миссии в `done`, `fsync` журнала.
6. Опубликовать `fblock.deleted` для фблока, который был перезаписан в оставшейся зоне (если в нём был непустой UUID).

Уведомления о самом перенесённом фблоке не публикуются — UUID сохраняется (см. 4.9 requirements).

#### 10.3.4. Финализация

Когда все миссии `done`:

1. Truncate основного хранилища до `N_new × fblock_size`.
2. Truncate каталога SSD до `N_new × 33` байт.
3. Обновить контрольную сумму каталога.
4. Удалить файл журнала.
5. Запустить Storage.

#### 10.3.5. Возобновление после сбоя

При следующем старте farc обнаруживает файл журнала и предлагает потребителю продолжить или откатить:

- **Продолжить:** перейти к выполнению миссий в статусе `pending` или `in_progress` (последние перезапускаются — повторное выполнение идемпотентно, целевой фблок ещё не помечен в каталоге как принадлежащий новому UUID).
- **Откатить:** удалить журнал, оставить Storage в исходной геометрии.

## 11. Импорт

Импорт копирует фблоки из одного Storage в другой. Storage-приёмник остановлен.

### 11.1. Алгоритм

1. **Открыть источник** в режиме чтения (либо через каталог SSD, либо сканированием — без блокировки на запись).
2. **Проверить совместимость:** `fblock_size` источника и приёмника должны совпадать.
3. **Для каждого `ready` фблока источника:**
   - Прочитать фблок целиком.
   - Найти UUID в каталоге приёмника:
     - Не найден — добавить как новый фблок (выбрать индекс по алгоритму записи). Скопировать UUID, времена, флаг `protected`, данные.
     - Найден, времена `begin/end` совпадают — пропустить (тот же фблок).
     - Найден, времена различаются — сгенерировать новый UUID, добавить как новый фблок.
4. После завершения — обновить каталог SSD приёмника, `fsync`.

### 11.2. Свойства

- Источник не модифицируется.
- Приёмник во время импорта не принимает обычную запись (он `stopped`).
- Импорт возобновляем при сбое: повторный запуск пропустит уже импортированные фблоки благодаря проверке UUID и времён.
