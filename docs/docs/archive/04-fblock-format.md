# Формат фблока

## 1. Общая структура

Фблок — это самодостаточная единица данных фиксированного размера. Состоит из двух частей:

- Заголовок (128 байт) — служебная информация, добавляемая библиотекой.
- Полезные данные — данные потребителя (размер равен fblock_size минус 128).

При передаче по сети или сохранении в отдельный файл фблок сохраняет полную структуру (заголовок плюс данные), что позволяет интерпретировать его без внешних метаданных.

## 2. Заголовок фблока

### 2.1. Структура заголовка

| Смещение | Размер | Поле | Описание |
|----------|--------|------|----------|
| 0 | 4 | magic | Сигнатура 0xFB1001 (little-endian) |
| 4 | 1 | version | Версия формата заголовка (текущая равна 1) |
| 5 | 1 | flags | Флаги состояния фблока (см. раздел 2.2) |
| 6 | 1 | type | Тип фблока (см. раздел 2.3) |
| 7 | 3 | reserved_1 | Выравнивание, заполнено нулями |
| 10 | 16 | uuid | UUID фблока (128 бит, RFC 4122 вариант 4) |
| 26 | 8 | expiry_time | Время окончания актуальности (Unix nanoseconds) |
| 34 | 8 | write_time | Время записи фблока на диск (Unix nanoseconds) |
| 42 | 8 | data_size | Размер полезных данных в байтах |
| 50 | 4 | header_checksum | CRC32C всех предыдущих полей заголовка |
| 54 | 8 | data_checksum | xxHash64 полезных данных |
| 62 | 66 | reserved_2 | Заполнено нулями для будущих расширений |

Итого: 128 байт.

### 2.2. Флаги (1 байт)

| Бит | Маска | Название | Описание |
|-----|-------|----------|----------|
| 0 | 0x01 | WRITTEN | Фблок содержит валидные данные, UUID корректен |
| 1 | 0x02 | IN_PROGRESS | Идёт запись, фблок не завершён, данные невалидны |
| 2 | 0x04 | BAD | Физическое повреждение, использовать нельзя |
| 3 | 0x08 | PROTECTED | Защита от перезаписи (read-only) |
| 4-7 | - | - | Резерв, должно быть 0 |

Сочетания флагов:

- Свободный фблок: все флаги 0, UUID нулевой.
- Записанный фблок: WRITTEN равен 1, IN_PROGRESS равен 0.
- В процессе записи: WRITTEN равен 0, IN_PROGRESS равен 1.
- Битый фблок: BAD равен 1 (может сочетаться с другими).
- Защищённый фблок: PROTECTED равен 1, WRITTEN равен 1.

Недопустимые сочетания:

- WRITTEN и IN_PROGRESS одновременно.
- PROTECTED без WRITTEN.

### 2.3. Тип фблока (1 байт)

| Значение | Название | Описание |
|----------|----------|----------|
| 0x00 | DATA | Обычный фблок с данными потребителя |
| 0x01 | INDEX | Фблок содержит копию индексов хранилища |
| 0x02-0xFF | - | Зарезервировано для будущих типов |

### 2.4. UUID

UUID генерируется библиотекой после успешной записи фблока. Используется UUID версии 4 (случайный).

Для индексных фблоков (тип 0x01) UUID имеет специальный формат:

- Байты с 0 по 7: нули.
- Байты с 8 по 15: номер копии в little-endian.

В стандартном текстовом представлении для копии 1: 00000000-0000-0000-0000-000000000001.

### 2.5. Временные метки

Все временные метки хранятся в формате Unix nanoseconds:

- Количество наносекунд с 00:00:00 UTC 1 января 1970 года.
- 64-битное целое без знака.
- Порядок байт: little-endian.

Поля:

- expiry_time — задаётся потребителем при заполнении буфера. Определяет, когда данные становятся неактуальными.
- write_time — заполняется библиотекой в момент начала записи фблока на диск.

### 2.6. Размер полезных данных

data_size — количество байт полезных данных, фактически записанных потребителем.

Может быть меньше максимального размера данных фблока (fblock_size минус 128). Оставшееся место в фблоке не используется (заполняется нулями при записи на диск).

### 2.7. Контрольные суммы

#### 2.7.1. Контрольная сумма заголовка

- Алгоритм: CRC32C (Castagnoli).
- Полином: 0x1EDC6F41.
- Начальное значение: 0xFFFFFFFF.
- Вычисляется по байтам с 0 по 49 заголовка (все поля до header_checksum).
- Порядок байт результата: little-endian.

#### 2.7.2. Контрольная сумма данных

- Алгоритм: xxHash64.
- Seed: 0 (по умолчанию).
- Вычисляется по первым data_size байтам полезных данных.
- Порядок байт результата: little-endian.

## 3. Полезные данные

### 3.1. Структура

Полезные данные — это непрерывная область размером data_size байт, следующая сразу за заголовком.

Физическое смещение данных в фблоке: 128 байт.

Максимальный размер полезных данных: fblock_size минус 128.

### 3.2. Выравнивание

Полезные данные не имеют требований к выравниванию. Потребитель может записывать данные произвольного формата.

### 3.3. Заполнение

Если data_size меньше чем fblock_size минус 128, оставшаяся часть фблока заполняется нулями при записи на диск. При чтении возвращаются только data_size байт.

## 4. Фчанки

### 4.1. Разбиение на фчанки

Фблок логически делится на фчанки — блоки фиксированного размера, используемые для операций ввода-вывода с проверкой.

Количество фчанков в фблоке: K равно fblock_size делить на fchunk_size.

Размер фчанка должен быть кратен размеру сектора диска (обычно 512 или 4096 байт) и нацело делить размер фблока.

### 4.2. Нумерация фчанков

Фчанки нумеруются от 0 до K минус 1.

Фчанок 0 содержит заголовок фблока и начало полезных данных.

Границы фчанков:

- Фчанок i занимает байты от i умножить на fchunk_size до (i плюс 1) умножить на fchunk_size минус 1 внутри фблока.

### 4.3. Атомарность фчанка

Фчанок является минимальной единицей, для которой гарантируется атомарность записи с точки зрения обнаружения ошибок:

- При записи каждый фчанок пишется, читается обратно и сравнивается.
- Если сравнение не совпадает, фчанок считается битым.
- Битый фчанок делает битым весь фблок.

## 5. Порядок байт

Все многобайтовые поля в заголовке фблока хранятся в порядке little-endian.

Это касается:

- magic (0xFB1001)
- uuid (как 16-байтовое значение)
- expiry_time (64 бита)
- write_time (64 бита)
- data_size (64 бита)
- header_checksum (32 бита)
- data_checksum (64 бита)

## 6. Валидация фблока при чтении

При чтении фблока (полном или частичном) библиотека выполняет следующие проверки:

1. Проверка сигнатуры:
   - magic должен быть равен 0xFB1001.
   - Если нет — фблок невалиден.

2. Проверка версии:
   - version должен быть равен 1.
   - Если нет — формат не поддерживается.

3. Проверка заголовка:
   - Вычислить CRC32C полей с 0 по 49.
   - Сравнить с header_checksum.
   - Если не совпадает — заголовок повреждён.

4. Проверка флагов:
   - Флаги должны быть в допустимом сочетании.
   - Если установлен BAD — фблок битый.
   - Если установлен IN_PROGRESS — фблок не завершён.

5. Проверка данных (только при полном чтении):
   - Вычислить xxHash64 первых data_size байт данных.
   - Сравнить с data_checksum.
   - Если не совпадает — данные повреждены.

## 7. Создание нового фблока

Алгоритм формирования заголовка при записи:

1. Заполнить поля:
   - magic равен 0xFB1001
   - version равен 1
   - flags равен IN_PROGRESS (0x02)
   - type равен DATA (0x00)
   - uuid нулевой (16 байт нулей)
   - expiry_time равен значению от потребителя
   - write_time равен 0 (будет заполнено позже)
   - data_size равен размеру данных от потребителя

2. Вычислить header_checksum (CRC32C).

3. Записать заголовок на диск.

4. После успешной записи данных:
   - Сгенерировать UUID.
   - Установить write_time равным текущему времени.
   - Сменить flags на WRITTEN (0x01).
   - Вычислить data_checksum (xxHash64).
   - Пересчитать header_checksum.
   - Записать обновлённый заголовок на диск.

## 8. Экспорт фблока

При экспорте фблока (для передачи по сети или сохранения в файл) сохраняется полный образ:

- Заголовок (128 байт).
- Полезные данные (data_size байт).

Оставшаяся часть фблока (нули до fblock_size) может не сохраняться, так как не содержит полезной информации. Размер экспортированного фблока: 128 плюс data_size байт.

При импорте такого фблока в другое хранилище:

- Если размер хранилища-приёмника допускает data_size, фблок записывается как есть.
- Если data_size больше максимального размера данных приёмника — ошибка.
