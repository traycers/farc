# ADR-006: Ленивая инициализация хранилища

**Дата:** 2026-04-24
**Статус:** Принято

## Контекст

Полная инициализация хранилища (запись заголовков во все фблоки) на диске 18 ТБ занимает порядка суток. Это неприемлемо при вводе нового диска в эксплуатацию.

## Варианты

### 1. Полная инициализация при создании
- ✔ После инициализации все фблоки сразу в состоянии `ready` — простота логики.
- ❌ Время инициализации пропорционально размеру диска. Для 18 ТБ — порядка суток.
- ❌ Диск не может принимать данные, пока инициализация не завершена.

### 2. Ленивая инициализация
- ✔ Инициализация = запись одного фблока (секунды).
- ✔ Диск начинает принимать данные немедленно.
- ❌ Необходимо новое состояние фблока (`uninitialized`) и логика его обработки.
- ❌ При старте нужен способ отличить `uninitialized` от повреждённых данных.

## Решение

Ленивая инициализация.

### Состояние `uninitialized`

Добавляется четвёртое базовое состояние фблока — `uninitialized`. Фблок в этом состоянии:
- Не содержит валидного заголовка (на диске мусор или нули).
- **Доступен для записи** Recorder'ом.
- **Не доступен для чтения** — попытка чтения данных возвращает ошибку.
- Определяется по отсутствию валидного `magic_prolog` в заголовке фблока или по нулевому байту `flags` в снимке каталога.

### Процесс инициализации

1. Создать файл нужного размера (или открыть блочное устройство).
2. Записать первый фблок (index=0) с заголовком, содержащим параметры хранилища: magic number, версия формата, размер фблока, размер фчанка, общее количество фблоков, начальный снимок индексов (все фблоки кроме нулевого — `uninitialized`).
3. Инициализация завершена. Хранилище готово к работе.

### Приоритет при выборе фблока для записи

Recorder при движении по кругу выбирает фблоки для записи в следующем порядке приоритета:

1. `uninitialized` — сначала заполняем весь диск.
2. Неактуальный `ready` — в состоянии `ready`, не `protected`, с истёкшим временем актуальности.

Это означает, что первый полный проход по диску заполняет все `uninitialized` фблоки. Со второго прохода работа идёт только с `ready` фблоками — обычный циклический режим.

### Определение `uninitialized` при старте

Есть два источника информации о том, что фблок `uninitialized`:

1. **По снимку каталога** (основной путь). В снимке каталога из свежайшего валидного фблока `uninitialized` фблоки помечены соответствующим значением в `flags[]` (кодирование `0` — нулевой байт, см. `02-storage-format.md`, раздел 6.2).
2. **По отсутствию `magic_prolog`** (fallback при сканировании). При полном сканировании хранилища — если заголовок фблока не содержит валидного `magic_prolog`, фблок считается `uninitialized`.

Два источника согласованы: свежевыделенный диск содержит нули, поэтому и `magic_prolog` отсутствует, и байт `flags` соответствующего фблока в снимке каталога равен 0 = `uninitialized`.

Это надёжно отличает неинициализированный фблок от повреждённого (`bad`): у `bad` заголовок был валиден (magic присутствует), но данные повреждены; у `uninitialized` заголовка никогда не было.

## Обоснование

Приоритет отдан скорости ввода диска в эксплуатацию. Усложнение логики (новое состояние, приоритет выбора) минимально и локализовано в `IndexManager`.
