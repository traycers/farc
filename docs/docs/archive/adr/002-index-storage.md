# ADR-002: Стратегия хранения индексов хранилища

**Дата:** 2026-04-15
**Статус:** Принято

## Контекст

Требуется хранить карту состояния фблоков (битые/защищённые). Рассматривались варианты централизованного хранения в начале/конце файла и распределённого хранения.

## Анализ вариантов

### 1. В конце хранилища
- ✔ Легко расширять.
- ❌ Медленный доступ (зона низких скоростей HDD).

### 2. В начале хранилища
- ✔ Быстрый доступ.
- ❌ Невозможно расширить том без сдвига данных.

### 3. Внешнее хранение (отдельный файл)
- ✔ После записи фблока не требует переходить к записи в другое место хранилища — индексы записываются на сторону.
- ❌ Риск рассинхронизации.
- ❌ При записи до заполнения диск могут изъять из системы, а индексы оставить в другом месте, что потребует очень внимательного слежения за синхронным перемещением хранилища и индексов.
- ✔ Если индексы переделать под формат системы внешнего хранения, можно делать поиск данных, не затрагивая работу хранилища.

### 4. В начале каждого фблока (распределённые индексы)
- ✔ Равномерный износ диска (нет горячей зоны перезаписи индексов).
- ✔ Данные о блоке физически рядом с блоком.
- ❌ Актуальные индексы после записи текущего фблока находятся в следующем фблоке.

### 5. В специально выделенных фблоках
- ✔ При изменении размера хранилища индексы в фблоках можно перемещать по хранилищу.
- ✔ При увеличении размера можно занимать несколько фблоков подряд.
- ❌ Сильное усложнение поиска фблоков с индексами при старте.

## Решение

Распределённое хранение индексов в заголовке каждого фблока.

**Обоснование:** хранение индексов в начале, конце или в выделенных фблоках подразумевает после записи каждого фблока возвращаться на конкретные места и перезаписывать индексы — это увеличивает количество записей в области индексов по сравнению с фблоками (неравномерный износ диска).

Хранение в начале и в конце подразумевает либо внутри фблоков, либо вне фблоков:
- Вне фблоков — нарушает расчёт смещений, ведь необходимо учитывать расположение и размер индексов.
- Внутри фблока — при большом размере фблока индексы могут занимать его малую часть, остальное место не используется.

Расположение всех данных в самом хранилище позволяет делать импорт и производить восстановление. Без индексов, которые были на стороне, это сделать нельзя из-за потери данных. Индексы на стороне можно хранить как необязательную копию только для ускорения поиска и ради возможности перенести поиск на другую машину для распределения нагрузки.
