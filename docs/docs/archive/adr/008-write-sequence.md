# ADR-008: Монотонный счётчик записей (write_sequence)

**Дата:** 2026-04-24
**Статус:** Принято

## Контекст

При старте Storage необходимо найти фблок с самым свежим снимком каталога (см. ADR-007, `03-storage-operations.md`). Для этого нужен критерий «свежести» каталога.

Первый кандидат — временная метка записи (`catalog_time` в Unix timestamp). Однако системные часы могут идти не монотонно:
- NTP-коррекция может сдвинуть время назад.
- Ручная установка времени администратором.
- Сбой RTC на сервере.

Если два фблока были записаны последовательно, но между ними часы сдвинулись назад, более старый фблок будет иметь большую временную метку. Алгоритм выбора «самого свежего» каталога по `catalog_time` выберет неправильный фблок. Это приведёт к восстановлению устаревших индексов и потенциально — к потере записанных данных.

## Варианты

### 1. Только временная метка
- ✔ Человекочитаемая информация в заголовке.
- ❌ Не гарантирует правильный порядок при сбоях системных часов.

### 2. Только монотонный счётчик
- ✔ Гарантия правильного порядка.
- ❌ Теряется информация о времени записи.

### 3. Временная метка + монотонный счётчик
- ✔ Гарантия правильного порядка (через счётчик).
- ✔ Временная метка сохраняется для диагностики и аудита.
- ❌ Дополнительные 8 байт в заголовке.

## Решение

В фиксированную часть пролога фблока добавляется поле `write_sequence` (uint64) — монотонно возрастающий счётчик записей в рамках одного Storage.

### Правила

1. Счётчик увеличивается на 1 при каждой записи фблока (включая `in_progress`, включая запись, которая потом пометится `bad`).
2. Начальное значение счётчика — 1 при инициализации Storage (первый записанный фблок получает `write_sequence = 1`).
3. Счётчик сохраняется между перезапусками: при старте Storage берётся максимальное значение `write_sequence` среди валидных фблоков, следующая запись получает значение `max + 1`.
4. `catalog_time` сохраняется в заголовке для аудита и диагностики, но не используется для определения порядка.
5. При поиске самого свежего каталога алгоритм выбирает фблок с максимальным `write_sequence`.

### Защита от переполнения

uint64 даёт 2⁶⁴ ≈ 1.8×10¹⁹ записей. При скорости 1000 записей в секунду (очень высокая для archive-системы) переполнение наступит через ≈ 585 миллионов лет. Практическое переполнение исключено.

## Обоснование

Правильный порядок восстановления индексов — критическое требование к надёжности хранилища. Монотонный счётчик решает эту задачу гарантированно, без зависимости от системных часов. 8 дополнительных байт в заголовке — ничтожная цена.
