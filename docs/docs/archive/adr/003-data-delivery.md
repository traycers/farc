# ADR-003: Выбор способа предоставления данных клиенту

**Дата:** 2026-04-16
**Статус:** Принято

## Контекст

Клиент вызывает API, нужен способ предоставления данных. Заранее не известен контекст использования данных из фблока.

## Сравнение

### 1. По HTTP отдавать одну часть фблока
Отдаём непрерывную часть фблока по смещению и размеру.

- ❌ Клиент должен знать индексы до чтения данных.
- ❌ Индексы в полном объёме занимают много места.
- ✔ Экономия трафика и IO.
- ❌ Вырождается в чтение всего фблока, когда нужны данные каналов, располагающихся в начале и конце фблока.
- ✔ Универсальный способ, не зависит от потребностей клиента.
- ✔ Простота в реализации.
- ❌ Отдаём только непрерывную часть.
- ✔ Возможность добавления стрима заполняющегося фблока.

### 2. По HTTP отдавать фблок целиком
- ✔ Универсальный способ, не зависит от потребностей клиента.
- ✔ У клиента максимум данных для решения его задачи.
- ❌ Чтение фблока с диска может занять ощутимое время (например, для размера фблока = 2 ГБ).
- ❌ Увеличение нагрузки на сеть.
- ✔ Простота в реализации.

### 3. По HTTP отдавать части фблока
Отдаём список непрерывных частей фблока по смещениям и размерам.

- ✔ Универсальный способ, не зависит от потребностей клиента.
- ✔ Экономия трафика и IO.
- ✔ Простота в реализации.
- ❌ Требует выбора контейнера для упаковки непрерывных частей.
- ❌ Клиент должен знать индексы до чтения данных.

### 4. RTSP-сервер
Предоставлять все данные потоковым образом.

- ✔ Сторонние решения могут использовать стандартные RTSP-клиенты, упрощение интеграции сторонних систем.
- ❌ Усложнения в реализации.
- ❌ Могут быть задержки вещания в момент смены фблоков, из которых берутся данные.
- ❌ Все типы данных нужно адаптировать под поточную модель.
- ✔ Если данные в фблоке располагаются в виде потоков, можно сделать универсальный сервер, но требует изучения протокола SDP на предмет добавления в него своих потоков данных.
- ❓ Требуется изучение передачи данных по зашифрованному каналу.
- ✔ Возможность добавления стрима заполняющегося фблока, live-режим.
- ✔ При смене параметров канала есть стандартный способ redirect.

### 5. WebSocket
Предоставлять все данные потоковым образом.

- ❌ Потребует реализации своего протокола, нет стандартных клиентов.
- ✔ Если отправлять fMP4, можно сделать воспроизведение в вебе.
- ✔ Подходит для live-режима.

## Решение

По HTTP отдавать фблок целиком.

**Обоснование:** самая простая и универсальная реализация. Далее легко адаптируется для передачи части фблока. Возможность использования HTTP-кэша вне farc.
