# ADR-007: Внешний каталог индексов (catalog)

**Дата:** 2026-04-24
**Статус:** Принято

## Контекст

Индексы хранилища распределены по заголовкам фблоков (ADR-002). При старте необходимо найти актуальные индексы — для этого нужно прочитать заголовки фблоков с диска. На HDD объёмом 18 ТБ полное сканирование заголовков занимает ощутимое время: каждый заголовок — это seek + чтение, умноженные на миллионы фблоков.

С появлением ленивой инициализации (ADR-006) проблема усугубляется: без полного сканирования неизвестно, где заканчиваются `uninitialized` фблоки и где находится последний записанный фблок с актуальными индексами.

В ADR-002 упоминалась возможность хранить «индексы на стороне как необязательную копию для ускорения поиска». Теперь это становится практической необходимостью.

## Варианты

### 1. Только распределённые индексы (текущий подход)
- ✔ Все данные в одном файле, самодостаточность.
- ❌ Долгий старт на больших дисках (полное сканирование заголовков).
- ❌ С ленивой инициализацией ещё медленнее — нужно определить границу `uninitialized`.

### 2. Индексы в начале/конце хранилища
- ✔ Быстрый доступ при старте.
- ❌ Нарушает `Index * FixedSize` (ADR-001).
- ❌ Горячая зона перезаписи (ADR-002).
- ❌ Усложняет расширение/сужение.

### 3. Внешний каталог на быстром диске
- ✔ Старт за секунды независимо от размера HDD.
- ✔ Не нарушает формат хранилища — каталог необязателен.
- ✔ Чтение каталога при поиске фблоков не затрагивает работу HDD.
- ❌ Необходимость синхронизации каталога с хранилищем.
- ❌ При потере каталога — fallback на полное сканирование (долго, но данные не теряются).

## Решение

Внешний каталог на отдельном диске (рекомендуется SSD).

### Структура каталога

Каталог — это **массив заголовков всех фблоков** одного Storage (или эквивалентная структура массивов — конкретная компоновка фиксируется в `02-storage-format.md`). Один файл каталога соответствует одному Storage.

Каталог содержит для каждого фблока:
- Состояние (`uninitialized` / `in_progress` / `ready` / `bad`).
- Флаг `protected`.
- UUID.
- Время начала и окончания актуальности.
- Время записи.
- Контрольная сумма заголовка.

### Жизненный цикл каталога

1. **Создание.** При инициализации Storage создаётся файл каталога. Все записи кроме нулевой помечены `uninitialized`.

2. **Обновление.** После каждой успешной записи фблока Recorder обновляет соответствующую запись каталога. Обновление включает:
   - Запись нового состояния фблока.
   - Актуальный снимок индексов (или его часть — зависит от формата).
   - `fsync` каталога для гарантии персистентности.

3. **Старт Storage.** `ConsistencyCheck` сначала пытается загрузить каталог. Если каталог валиден — индексы берутся из него (секунды). Если каталог отсутствует или повреждён — fallback на сканирование заголовков фблоков на HDD (долго).

4. **Верификация.** При загрузке из каталога `ConsistencyCheck` проверяет контрольную сумму каталога и может выборочно проверить несколько заголовков на HDD для подтверждения согласованности.

### Необязательность

Каталог — оптимизация, а не обязательный компонент:
- При потере файла каталога данные хранилища не теряются.
- Каталог можно пересоздать из хранилища в любой момент (полным сканированием).
- Storage без каталога полностью функционален, только старт медленнее.

### Размещение

- Один файл каталога на один Storage.
- Рекомендуемое размещение — на SSD, отдельном от HDD с данными.
- Путь к каталогу задаётся при создании Storage и хранится в конфигурации Archive.

## Обоснование

Внешний каталог решает проблему долгого старта без изменения формата хранилища и без нарушения принципа самодостаточности (данные в хранилище остаются полными и восстановимыми без каталога). Размещение на отдельном диске дополнительно разгружает HDD от операций поиска при чтении индексов.
