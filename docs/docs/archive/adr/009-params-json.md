# ADR-009: Сериализация параметров пролога в JSON

**Дата:** 2026-04-24
**Статус:** Принято

## Контекст

В прологе фблока хранится набор параметров Storage: размер фчанка, режим записи и другие, которые могут быть добавлены в будущем. Необходимо выбрать формат сериализации этих параметров.

Требования к формату:
- **Forward compatibility** — новые версии библиотеки могут добавлять параметры, старые версии должны уметь читать хранилище, пропуская незнакомые параметры.
- **Самоописываемость** — имя и тип поля видны при чтении, без внешней схемы.
- **Небольшой объём** — параметры занимают не более нескольких сотен байт на фблок.
- **Поддержка байтовых последовательностей** — для UUID, хешей, бинарных идентификаторов.

Параметры читаются при старте Storage и кэшируются в памяти. Скорость парсинга — не критичный фактор.

## Варианты

### 1. Самописный формат узлов (Parent & Sibling vectors)
- ✔ Минимальный оверхед.
- ✔ Нативная поддержка байтовых массивов без кодирования.
- ✔ Forward compatibility встроена (пропуск узла по size).
- ❌ Нужно реализовать парсер самим.
- ❌ Нет готовых инструментов для просмотра и редактирования.

### 2. JSON
- ✔ Универсальный, поддерживается везде.
- ✔ Forward compatibility естественная (пропустить незнакомый ключ).
- ✔ Человекочитаемый — удобен для диагностики и CLI-утилит.
- ✔ Готовые парсеры на всех языках.
- ❌ Нет нативной поддержки байтовых последовательностей — требуется кодирование (base64, hex).
- ❌ Больше оверхеда, чем у бинарных форматов.

### 3. CBOR / MessagePack
- ✔ Бинарный, компактный.
- ✔ Нативная поддержка байтовых данных.
- ❌ Дополнительная зависимость.
- ❌ Менее удобно для диагностики.

## Решение

Параметры пролога сериализуются в **JSON** (UTF-8, без BOM).

### Обоснование

- Объём параметров невелик (десятки-сотни байт), оверхед JSON приемлем.
- Удобство диагностики важнее компактности — заголовки фблоков придётся читать и анализировать инструментами восстановления, а читаемый JSON упрощает эту работу.
- Для байтовых последовательностей, если понадобятся, используется hex-строка или base64 — решается при введении конкретного параметра.
- Форматы CBOR/MessagePack добавляют зависимость без существенной выгоды при таком объёме данных.

### Область применения

JSON используется **только** для параметров пролога. Каталог сериализуется в плотном бинарном формате (структура массивов) и не использует JSON — объём каталога на миллионы фблоков делает JSON-оверхед неприемлемым.

### Эволюция

Форматные изменения отслеживаются через `format_version` в фиксированной части пролога. При необходимости будущие версии формата могут заменить JSON на более компактное представление. Текущая версия (v1) использует JSON.

### Ограничения

- Запрещены комментарии и trailing commas — строгий JSON по RFC 8259.
- Порядок ключей не определён, парсер не должен полагаться на него.
- Числа — целые (int64) или строки (для значений вне диапазона int64).

## Пример

```json
{
  "fchunk_size": 4194304,
  "recording": {
    "mode": "cyclic"
  }
}
```
