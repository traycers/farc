# ADR-004: Выбор способа чтения

**Дата:** 2026-04-17
**Статус:** Принято

## Контекст

Потребителю требуется конкретный фрагмент фблока (например, GOP для создания JPEG) для реализации API.

## Сравнение

### 1. Читаем весь фблок
- ❌ Очень долгое чтение при большом размере фблока.
- ❌ Риск блокировки записи в момент чтения фблока.
- ❌ Незачем читать всё, когда необходима только часть фблока.
- ✔ Нет необходимости во внешних индексах для определения места чтения.

### 2. Читаем непрерывную часть фблока
- ❌ Формат фблока должен иметь возможность чтения без парсинга всей структуры фблока.
- ✔ Потребитель может запрашивать только нужные данные на основе индексов самих фблоков, располагающихся на стороне.
- ❌ Индексы самих фблоков занимают много места; для его экономии нужно удалять информацию из индексов, а это приводит к тому, что потребитель может предоставить только приблизительные смещения.
- ❌ Вырождается в чтение всего фблока, когда данные располагаются в его крайних местах.

### 3. Читаем части фблока в несколько этапов
Потребитель по индексам хранилища находит нужный ему фблок; читает из фблока его индексы; определяет смещения и размеры нужных ему данных; StorageEngine читает несколько запрошенных частей.

- ✔ Нет нужды в индексах фблоков на стороне.
- ❌ Сложности в реализации, ведь команды на чтение приходят в FIFO-очередь, а это значит: StorageEngine после чтения индексов ждёт, пока потребитель решит, какие данные ему нужны; потребитель может зависнуть, тогда заблокируется запись.
- ✔ Читаем только необходимые данные.

## Решение

Читаем части фблока в несколько этапов.

**Обоснование:** важно обойтись без индексов фблоков на стороне из-за их огромного размера. Да, придётся FIFO-очередь заменить на иное решение. Можно разделить по потокам: StorageEngine отдаёт индексы и в очередь ставит следующий этап, т.е. команда на чтение усложняется — из простого «дай N байтов» в чтение из нескольких этапов.
