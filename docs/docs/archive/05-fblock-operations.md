# Алгоритмы работы с фблоком

## 1. Общие положения

Фблок — это логическая единица данных, с которой работает потребитель. Библиотека предоставляет API для заполнения фблоков данными и последующей записи на диск.

Все операции с фблоком делятся на две фазы:

- Фаза заполнения: потребитель получает буфер, заполняет его данными.
- Фаза записи: потребитель передаёт заполненный буфер библиотеке для записи на диск.

## 2. Получение буфера для заполнения

### 2.1. API

Потребитель вызывает функцию get_buffer.

Возвращает:

- Указатель на буфер (или handle).
- Размер буфера в байтах.
- Статус загрузки системы.

### 2.2. Статусы загрузки

| Статус | Описание | Рекомендация потребителю |
|--------|----------|--------------------------|
| NORMAL | Очередь записи в норме (менее 70 процентов) | Работа без ограничений |
| WARNING | Очередь приближается к переполнению (от 70 до 90 процентов) | Снизить темп наполнения |
| BACKPRESSURE | Очередь переполнена (более 90 процентов) | Применить стратегию деградации |
| NO_BUFFER | Нет свободных буферов | Повторить попытку позже |

### 2.3. Алгоритм get_buffer

1. Заблокировать BufferPool.
2. Если есть свободный буфер:
   - Взять буфер из пула.
   - Пометить как «заполняется».
   - Уменьшить счётчик свободных буферов.
   - Определить статус загрузки по текущему размеру WriteQueue.
   - Разблокировать BufferPool.
   - Вернуть буфер и статус.
3. Если свободных буферов нет:
   - Разблокировать BufferPool.
   - Вернуть статус NO_BUFFER.

### 2.4. Размер буфера

Размер предоставляемого буфера равен fblock_size минус 128 байт (размер заголовка).

Потребитель может записать в буфер меньше данных, чем размер буфера. Фактический размер записанных данных будет передан при вызове submit_buffer.

## 3. Заполнение буфера

### 3.1. Обязанности потребителя

Потребитель должен:

1. Заполнить буфер данными.
2. Определить время окончания актуальности данных (expiry_time).
3. Определить время начала актуальности данных (опционально, для внешнего использования).

Библиотека не анализирует содержимое буфера и не модифицирует данные.

### 3.2. Время окончания актуальности

expiry_time — это момент времени, после которого данные считаются неактуальными и могут быть перезаписаны.

Форматы задания:

- Абсолютное время: Unix timestamp в наносекундах.
- Относительное время: длительность хранения в секундах (библиотека преобразует в абсолютное).
- Нулевое значение: данные актуальны бессрочно (не будут перезаписаны автоматически).

Если потребитель не указывает expiry_time, используется глобальное значение по умолчанию из конфигурации хранилища.

### 2.3. Алгоритм submit_buffer (синхронный режим)

**Входные параметры:**

- handle буфера.
- data_size — фактический размер записанных данных.
- expiry_time — время окончания актуальности.

**Алгоритм:**

1. Проверить, что handle валиден и буфер находится в состоянии «заполняется».
2. Проверить, что data_size не превышает размер буфера.
3. Сформировать структуру PendingWrite:
   - Указатель на буфер.
   - data_size.
   - expiry_time.
   - Временную метку постановки в очередь (для метрик).
4. Заблокировать WriteQueue.
5. Поместить PendingWrite в очередь.
6. Разблокировать WriteQueue.
7. Сигнализировать потоку записи о наличии работы.
8. Если вызван синхронный вариант:
   - Заблокироваться до завершения записи этого буфера.
   - После записи получить UUID из результата.
   - Вернуть UUID потребителю.
9. Если вызван асинхронный вариант:
   - Вернуть управление немедленно.
   - UUID будет доставлен через EventBus позже.

### 4.2. Обработка очереди потоком записи

Поток записи работает в цикле:

1. Ожидать появления элементов в WriteQueue.
2. Извлечь PendingWrite из очереди (FIFO).
3. Вызвать алгоритм записи фблока (см. 03-storage-operations.md, раздел 2).
4. Если запись успешна:
   - Получить сгенерированный UUID.
   - Если это был синхронный вызов — разблокировать ожидающего потребителя и передать UUID.
   - Если асинхронный — отправить событие NEW_FBLOCK через EventBus.
   - Освободить буфер (вернуть в BufferPool).
5. Если запись не удалась (нет места, диск недоступен):
   - Если синхронный вызов — вернуть ошибку потребителю.
   - Если асинхронный — отправить событие об ошибке.
   - Буфер остаётся занятым (потребитель может повторить попытку или освободить буфер).

## 5. Освобождение буфера без записи

### 5.1. API

Потребитель может вызвать release_buffer, если передумал сохранять данные.

### 5.2. Алгоритм release_buffer

1. Заблокировать BufferPool.
2. Проверить, что буфер в состоянии «заполняется».
3. Очистить буфер (опционально, для безопасности).
4. Пометить буфер как «свободен».
5. Увеличить счётчик свободных буферов.
6. Разблокировать BufferPool.

## 6. Чтение данных из записанного фблока

### 6.1. Полное чтение

**Входные параметры:**

- UUID фблока.

**Алгоритм:**

1. Вызвать внутренний метод хранилища для чтения фблока по UUID.
2. Получить данные и размер.
3. Вернуть потребителю указатель на данные и размер.

### 6.2. Частичное чтение

**Входные параметры:**

- UUID фблока.
- offset — смещение от начала данных.
- size — количество байт для чтения.

**Алгоритм:**

1. Вызвать внутренний метод хранилища для частичного чтения.
2. Получить запрошенный фрагмент данных.
3. Вернуть потребителю.

### 6.3. Потоковое чтение

Для больших фблоков потребитель может читать данные порциями, не загружая весь фблок в память.

**Входные параметры:**

- UUID фблока.

**Алгоритм:**

1. Открыть сессию чтения (получить handle).
2. Вызвать seek для позиционирования.
3. Циклически вызывать read для чтения порций.
4. Закрыть сессию.

## 7. Экспорт фблока

### 7.1. Назначение

Экспорт используется для:

- Передачи фблока по сети.
- Сохранения в отдельный файл.
- Переноса на другое хранилище.

### 7.2. Алгоритм экспорта

**Входные параметры:**

- UUID фблока.
- Callback для записи данных (или файловый дескриптор).

**Алгоритм:**

1. Найти фблок по UUID.
2. Прочитать заголовок (128 байт).
3. Передать заголовок получателю.
4. Прочитать data_size байт полезных данных.
5. Передать данные получателю.

Размер экспортированного образа: 128 плюс data_size байт.

## 8. Импорт фблока

### 8.1. Назначение

Импорт используется для:

- Приёма фблока по сети.
- Загрузки из файла.
- Восстановления данных утилитой восстановления.

### 8.2. Алгоритм импорта

**Входные параметры:**

- Источник данных (callback или файловый дескриптор).
- Опционально: новый expiry_time (если нужно переопределить).

**Алгоритм:**

1. Получить буфер через get_buffer.
2. Прочитать 128 байт заголовка из источника.
3. Проверить magic и version заголовка.
4. Извлечь data_size из заголовка.
5. Проверить, что data_size не превышает размер буфера.
6. Прочитать data_size байт данных из источника в буфер.
7. Проверить data_checksum (если требуется).
8. Определить expiry_time:
   - Если передан новый expiry_time — использовать его.
   - Иначе использовать expiry_time из заголовка.
9. Вызвать submit_buffer для записи импортированного фблока.
10. Вернуть UUID нового фблока потребителю.

## 9. Метрики фблока

### 9.1. Сбор метрик

При операциях с фблоком собираются следующие метрики:

**Метрики заполнения:**

- Время от get_buffer до submit_buffer.
- Размер записанных данных (data_size).

**Метрики очереди:**

- Время ожидания в WriteQueue.
- Количество буферов в очереди на момент постановки.

**Метрики записи:**

- Время записи фблока на диск.
- Количество повторов записи фчанков.

### 9.2. Доступ к метрикам

Метрики доступны через API библиотеки:

- get_fblock_stats() — статистика по конкретному фблоку (если известен UUID).
- get_queue_stats() — текущее состояние очереди.
- get_pool_stats() — состояние пула буферов.

## 10. Уведомления

### 10.1. События, связанные с фблоком

| Событие | Когда отправляется | Данные |
|---------|-------------------|--------|
| NEW_FBLOCK | После успешной записи | UUID, write_time, data_size |
| FBLOCK_DELETED | Перед перезаписью | UUID удаляемого фблока |
| WRITE_ERROR | При ошибке записи | UUID (если был), код ошибки |

### 10.2. Подписка на события

Потребитель регистрирует callback-функции через API:

- subscribe_new_fblock(callback)
- subscribe_fblock_deleted(callback)
- subscribe_write_error(callback)

Callback вызывается асинхронно из потока уведомлений библиотеки.

## 11. Защита фблока от перезаписи

### 11.1. Установка защиты

**Входные параметры:**

- UUID фблока.

**Алгоритм:**

1. Заблокировать IndexManager.
2. Найти запись индекса по UUID.
3. Если фблок не найден или битый — ошибка.
4. Установить флаг PROTECTED в индексах.
5. Разблокировать IndexManager.
6. Обновить заголовок фблока на диске (установить флаг PROTECTED).
7. Вызвать синхронизацию индексов.

### 11.2. Снятие защиты

**Входные параметры:**

- UUID фблока.

**Алгоритм:**

1. Заблокировать IndexManager.
2. Найти запись индекса по UUID.
3. Снять флаг PROTECTED в индексах.
4. Разблокировать IndexManager.
5. Обновить заголовок фблока на диске (снять флаг PROTECTED).
6. Вызвать синхронизацию индексов.

## 12. Ручное удаление фблока

### 12.1. API

Потребитель может явно удалить фблок, даже если его expiry_time ещё не наступил.

**Входные параметры:**

- UUID фблока.

**Алгоритм:**

1. Заблокировать IndexManager.
2. Найти запись индекса по UUID.
3. Если фблок защищён — ошибка (сначала снять защиту).
4. Пометить фблок как свободный:
   - Обнулить UUID.
   - Снять все флаги, кроме BAD и PROTECTED (их не может быть по условию).
   - Обнулить expiry_time и write_time.
5. Разблокировать IndexManager.
6. Вызвать синхронизацию индексов.
7. Отправить событие FBLOCK_DELETED.

## 13. Обработка ошибок

### 13.1. Ошибки при заполнении

| Ошибка | Причина | Действие |
|--------|---------|----------|
| NO_BUFFER | Все буферы заняты | Повторить позже |
| BUFFER_TOO_SMALL | data_size превышает размер буфера | Уменьшить данные |
| INVALID_HANDLE | Передан невалидный handle | Ошибка программирования |

### 13.2. Ошибки при записи

| Ошибка | Причина | Действие |
|--------|---------|----------|
| NO_SPACE | Нет свободных фблоков | Авария, требуется вмешательство |
| DISK_ERROR | Ошибка ввода-вывода | Авария, диск недоступен |
| BAD_FBLOCK | Фблок стал битым при записи | Автоматический переход на другой фблок |

### 13.3. Ошибки при чтении

| Ошибка | Причина | Действие |
|--------|---------|----------|
| UUID_NOT_FOUND | Фблок удалён или перезаписан | Данные утеряны |
| BAD_FBLOCK | Фблок помечен как битый | Данные недоступны |
| CHECKSUM_MISMATCH | Данные повреждены | Данные недоступны |
