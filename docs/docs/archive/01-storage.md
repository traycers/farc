### 3.2. Флаги (1 байт)

| Бит | Флаг | Описание |
|-----|------|----------|
| 0 | записан | Фблок содержит данные (UUID валиден) |
| 1 | в процессе | Идёт запись, фблок не завершён |
| 2 | битый | Физическое повреждение, использовать нельзя |
| 3 | заблокирован | Защита от перезаписи (read-only) |
| 4-7 | резерв | Для будущего |

### 3.3. Типы фблоков

- **0x00 — данные потребителя:** обычный фблок с полезными данными
- **0x01 — индексы:** фблок содержит копию индексов всего хранилища

UUID для индексных фблоков имеет специальный формат:
- Младшие 8 байт = 0
- Старшие 8 байт = номер копии (1, 2, 3...)

Пример: `00000000-0000-0000-0000-000000000001` для первой копии.

## 5. Расширение хранилища

Расширение происходит без остановки записи.

**Исходное состояние:**

**Шаг 1: Добавление места**

**Шаг 2: Создание новых индексных фблоков**

**Шаг 3: Обновление суперблока**

**Шаг 4: Продолжение работы**

**Время расширения:** только на запись индексов в новые фблоки + обновление суперблока = миллисекунды.

## 6. Процесс записи фблока (с точки зрения хранилища)

1. **Выбор целевого фблока:**
   - Циклически обходим фблоки, начиная с последнего записанного
   - Пропускаем битые и заблокированные
   - Если фблок записан, проверяем `текущее_время >= время_окончания_актуальности`

2. **Пометка "в процессе":**
   - В заголовке целевого фблока выставляем флаг `в процессе = 1`
   - Записываем заголовок (только первую копию? или все?)

3. **Запись по фчанкам с verify:**
   - Для каждого фчанка:
     - Запись на диск
     - Чтение обратно
     - Сравнение с оригиналом
     - При несовпадении — повтор (до 3 раз)
     - Если всё равно не совпадает — весь фблок считается битым

4. **При успехе:**
   - Генерируем UUID
   - Заполняем заголовок: UUID, время_записи, время_окончания, размер_данных
   - Снимаем флаг `в процессе`, выставляем флаг `записан`
   - Записываем финальный заголовок

5. **Обновление индексов:**
   - В памяти: обновляем запись для этого индекса (UUID, время_окончания, время_записи, флаги)
   - Для каждой копии индексов (из списка в суперблоке):
     - Читаем фблок с индексами
     - Обновляем в нём нужную запись (по индексу фблока)
     - Пересчитываем контрольную сумму
     - Записываем фблок целиком (с verify)

6. **Уведомление потребителя:**
   - Отправляем событие о новом фблоке (UUID, время_записи)

**Время записи фблока:** ~25-30 секунд (для 512 МБ с verify)

## 7. Восстановление после сбоя

При старте системы:

1. **Читаем суперблок** (первый сектор)
2. **Проверяем контрольную сумму** суперблока
3. **Получаем список индексных фблоков** из суперблока
4. **Читаем все копии индексов:**
   - Для каждого индексного фблока читаем его данные
   - Проверяем контрольную сумму всей копии
   - Запоминаем время записи копии (хранится в начале данных)
5. **Выбираем самую свежую целую копию** (по времени записи)
6. **Загружаем индексы в память**
7. **Сканируем фблоки с флагом "в процессе":**
   - Читаем заголовки всех фблоков (быстрое сканирование 128-байтовых заголовков)
   - Находим фблоки с флагом `в процессе = 1`
   - Помечаем их в индексах как `битые`
   - Обновляем копии индексов (записываем в индексные фблоки)

Время восстановления:
- Чтение суперблока: 1 seek
- Чтение копий индексов: 2-3 seeks + 1-2 МБ данных = **~40 мс**
- Сканирование заголовков: 37,000 × 128 байт = 4.7 МБ, но с seek-ами = **~2-3 минуты** (можно делать в фоне)

**Оптимизация:** сканирование заголовков для поиска "в процессе" можно делать в фоновом потоке, а основную работу начинать сразу после загрузки индексов.

## 8. Размещение на диске (пример для 20 ТБ)

Параметры:
- Размер фблока: 512 МБ
- Количество фблоков: 37,252
- Количество копий индексов: 4 (2 в начале, 2 в конце)

**Объём под индексы:** 4 × 512 МБ = 2 ГБ (0.01% от 20 ТБ)
