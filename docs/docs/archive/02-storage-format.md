# Бинарный формат фблока

## 1. Назначение документа

Документ описывает бинарный формат фблока: расположение и размеры полей, порядок байтов, контрольные суммы, правила парсинга. Является спецификацией для реализации чтения и записи фблоков.

Документ **не описывает**:
- Концептуальную модель данных, компоненты и их ответственность — см. `01-storage.md`.
- Алгоритмы операций (сканирование при старте, выбор фблока для записи, расширение, сужение) — см. `03-storage-operations.md`.

## 2. Общие соглашения

- Порядок байтов: **Little Endian** (LE) для всех многобайтовых целых.
- Временные метки: **Unix timestamp в наносекундах**, uint64.
- Контрольные суммы: **CRC32** (IEEE 802.3, полином `0x04C11DB7`).

## 3. Общая структура фблока

Фблок — непрерывная область фиксированного размера на диске. Состоит из шести секций:

```
┌─────────────────────────────────────────┐
│ Пролог                                  │  Фиксированная часть + параметры
├─────────────────────────────────────────┤
│ Каталог                                 │  Снимок состояния всех фблоков Storage
├─────────────────────────────────────────┤
│ Контрольные суммы заголовка             │  CRC32 × 3
├─────────────────────────────────────────┤
│ Данные                                  │  Payload (фчанки)
├─────────────────────────────────────────┤
│ TOC                                     │  Оглавление данных (непрозрачно для Storage)
├─────────────────────────────────────────┤
│ Эпилог                                  │  CRC32 данных, CRC32 TOC, размер TOC
└─────────────────────────────────────────┘
```

Чтение фблока выполняется с двух сторон: пролог и каталог — с начала, эпилог — с конца.

## 4. Формат узла дерева

Параметры и каталог используют единый формат сериализации — упакованное дерево на основе Parent & Sibling vectors. Данные представляются как последовательность узлов, каждый из которых содержит ссылки на родителя и левого соседа, семантический идентификатор и тип данных.

### 4.1. Структура узла

| Смещение | Размер | Тип    | Поле       | Описание |
|----------|--------|--------|------------|----------|
| 0        | 2      | uint16 | parent_id  | Индекс родительского узла. |
| 2        | 2      | uint16 | sibling_id | Индекс левого соседа. |
| 4        | 2      | uint16 | id         | Семантический идентификатор узла. Определяет *что* хранит узел. |
| 6        | 1      | uint8  | type       | Тип данных. Определяет *как* читать data. |
| 7        | 4      | uint32 | size       | Размер поля `data` в байтах. |

**Заголовок узла: 11 байт.** Поле `data` размером `size` байт следует сразу за заголовком. При `size = 0` поле `data` отсутствует.

### 4.2. Типы данных

| type | Имя       | Допустимые размеры | Описание |
|------|-----------|--------------------|----------|
| 0x00 | master    | 0                  | Контейнер. Не содержит данных, дети определяются по parent_id. |
| 0x01 | uint      | 1, 2, 4, 8         | Unsigned integer, Little Endian. |
| 0x02 | int       | 1, 2, 4, 8         | Signed integer, Little Endian, дополнение до двух. |
| 0x03 | float     | 4, 8               | IEEE 754, 32-bit или 64-bit. |
| 0x04 | bytes     | 0..2³²             | Сырые байты. |
| 0x05 | utf8      | 0..2³²             | Строка UTF-8. |
| 0x06 | uuid      | 16                 | UUID (RFC 4122). |
| 0x07 | timestamp | 8                  | Unix timestamp в наносекундах, uint64. |

### 4.3. Правила навигации

- **Корневой узел** (индекс 0): `parent_id = 0`, `sibling_id = 0`.
- **Самый левый ребёнок** (нет левого соседа): `sibling_id` равен собственному индексу узла.
- **Остальные узлы**: `sibling_id` равен индексу левого соседа.

### 4.4. Правило парсинга

Парсер читает узлы последовательно. Узел с незнакомым `id` пропускается по полю `size`. При этом `type` позволяет корректно отобразить данные незнакомого узла без знания его семантики. Это обеспечивает прямую совместимость: новые узлы добавляются в любую позицию среди siblings без изменения формата.

### 4.5. Пример

Дерево параметров с корнем и тремя узлами, где `RECORDING` содержит дочерний `MODE`:

```
Node 0: parent=0 sibling=0 id=ROOT        type=master  size=0
Node 1: parent=0 sibling=1 id=FCHUNK_SIZE type=uint    size=4  data=[00 40 00 00]
Node 2: parent=0 sibling=1 id=RECORDING   type=master  size=0
Node 3: parent=2 sibling=3 id=REC_MODE    type=uint    size=1  data=[01]
```

- Node 1: `sibling=1` — сам на себя, первый ребёнок ROOT.
- Node 2: `sibling=1` — левый сосед Node 1.
- Node 3: `sibling=3` — сам на себя, первый ребёнок RECORDING.

## 5. Пролог

Пролог — начальная секция фблока. Содержит фиксированную часть и параметры хранилища.

### 5.1. Фиксированная часть

Фиксированная часть пролога **никогда не меняет свою структуру**. Поля, их размеры и смещения постоянны для всех версий формата.

| Смещение | Размер | Тип    | Поле                        | Описание |
|----------|--------|--------|-----------------------------|----------|
| 0        | 4      | uint32 | magic                       | Магическое число. Идентифицирует блок как фблок farc. |
| 4        | 8      | uint64 | catalog_time                | Время создания снимка каталога в Unix timestamp наносекундах. Используется для определения самого свежего каталога при сканировании. |
| 12       | 8      | uint64 | fblock_size                 | Размер фблока в байтах. |
| 20       | 4      | uint32 | params_size                 | Размер секции параметров в байтах. |
| 24       | 4      | uint32 | catalog_entry_count         | Количество записей в каталоге (равно количеству фблоков в Storage). |
| 28       | 4      | uint32 | catalog_size                | Размер каталога в байтах. |

**Итого: 32 байта.**

### 5.2. Параметры

Параметры хранилища следуют сразу за фиксированной частью. Сериализованы в формате узлов дерева (раздел 4). Размер определяется полем `params_size` фиксированной части.

#### Семантические идентификаторы параметров

| id     | Имя          | type      | Описание |
|--------|--------------|-----------|----------|
| 0x0000 | ROOT         | master    | Корневой узел. |
| 0x0001 | FCHUNK_SIZE  | uint      | Размер фчанка в байтах. |
| 0x0002 | RECORDING    | master    | Группа параметров записи. |
| 0x0003 | REC_MODE     | uint      | Режим записи: 0x01 = циклический, 0x02 = до заполнения. |

Пример для фчанка 4 МБ, циклическая запись:

```
Node 0: parent=0 sibling=0 id=0x0000 type=master size=0
Node 1: parent=0 sibling=1 id=0x0001 type=uint   size=4  data=[00 00 40 00]
Node 2: parent=0 sibling=1 id=0x0002 type=master size=0
Node 3: parent=2 sibling=3 id=0x0003 type=uint   size=1  data=[01]
```

Правила расширения: новые параметры — новые узлы с новыми id. Парсер пропускает узлы с незнакомым id по полю `size`, при этом поле `type` позволяет корректно отобразить данные.

### 5.3. Вычисление смещений после пролога

Начало каталога: `32 + params_size`.

## 6. Каталог

Каталог — снимок состояния всех фблоков Storage на момент начала записи текущего фблока. Расположен сразу после пролога. Сериализован в формате узлов дерева (раздел 4) с плоской структурой: все узлы данных — дети корня.

### 6.1. Структура каталога

Каждый узел содержит массив значений одного поля для всех фблоков. Количество элементов в каждом массиве равно `catalog_entry_count`.

```
Node 0: parent=0 sibling=0 id=ROOT   type=master    size=0
Node 1: parent=0 sibling=1 id=UUID   type=uuid      size=N×16
Node 2: parent=0 sibling=1 id=FLAGS  type=bytes     size=N×1
Node 3: parent=0 sibling=2 id=BEGIN  type=timestamp  size=N×8
Node 4: parent=0 sibling=3 id=END    type=timestamp  size=N×8
Node 5: parent=0 sibling=4 id=CHAN64 type=bytes     size=N×128
```

### 6.2. Семантические идентификаторы каталога

| id     | Имя    | type      | Размер элемента | Описание |
|--------|--------|-----------|-----------------|----------|
| 0x0000 | ROOT   | master    | 0               | Корневой узел. |
| 0x0001 | UUID   | uuid      | 16 байт         | UUIDv4 фблока. Нулевой UUID — фблок не содержит данных. |
| 0x0002 | FLAGS  | bytes     | 1 байт          | Состояние и флаги фблока (см. 6.3). |
| 0x0003 | BEGIN  | timestamp | 8 байт          | Время начала актуальности фблока (ns). |
| 0x0004 | END    | timestamp | 8 байт          | Время окончания актуальности фблока (ns). |
| 0x0005 | CHAN64 | bytes     | 128 байт        | Массив номеров каналов (64 × uint16). Заполняется слева направо, неиспользованные слоты = 0. Нумерация каналов начинается с 1. |

Пространства семантических идентификаторов параметров и каталога **независимы**: id `0x0001` в параметрах (FCHUNK_SIZE) и id `0x0001` в каталоге (UUID) — разные узлы. Контекст определяется секцией.

### 6.3. Кодирование FLAGS

| Бит | Маска | Значение | Описание |
|-----|-------|----------|----------|
| 0-1 | 0x03  | 0        | `ready` |
|     |       | 1        | `in_progress` |
|     |       | 2        | `bad` |
|     |       | 3        | зарезервировано |
| 2   | 0x04  | 1        | `protected` |
| 3-7 |       |          | зарезервированы, должны быть 0 |

### 6.4. Размер каталога

Для стандартного набора узлов (ROOT, UUID, FLAGS, BEGIN, END, CHAN64):

```
Размер = 6 × 11 + N × (16 + 1 + 8 + 8 + 128) = 66 + N × 161 байт
```

где N — количество фблоков (`catalog_entry_count`).

### 6.5. Свойство снимка

Снимок каталога в заголовке фблока `K` отражает состояние Storage на момент начала записи `K`:
- Сам `K` в собственном снимке помечен `in_progress`.
- Предыдущий успешно записанный фблок помечен `ready`.

Подтверждение «фблок `M` успешно записан» попадает на диск в каталоге следующего записанного фблока `M+1`.

## 7. Контрольные суммы заголовка

Три контрольных суммы расположены последовательно сразу после каталога. Позволяют определить, какая именно часть заголовка повреждена.

| Смещение от начала секции | Размер | Тип    | Поле               | Покрывает |
|---------------------------|--------|--------|---------------------|-----------|
| 0                         | 4      | uint32 | crc32_fixed         | Фиксированная часть пролога (байты 0..31). |
| 4                         | 4      | uint32 | crc32_params        | Параметры целиком. |
| 8                         | 4      | uint32 | crc32_catalog       | Каталог целиком. |

**Итого: 12 байт.**

Смещение секции от начала фблока: `32 + params_size + catalog_size`.

### 7.1. Диагностика повреждений

| crc32_fixed | crc32_params | crc32_catalog | Интерпретация |
|-------------|--------------|---------------|---------------|
| ✓           | ✓            | ✓             | Заголовок целый. |
| ✓           | ✓            | ✗             | Параметры доступны, снимок каталога потерян. |
| ✓           | ✗            | ✓             | Каталог доступен, параметры повреждены. |
| ✓           | ✗            | ✗             | Только фиксированная часть валидна. |
| ✗           | —            | —             | Фблок нечитаем. |

## 8. Данные

Данные (payload) расположены после контрольных сумм заголовка. Записываются и верифицируются пофчанково (write-verify). Внутренняя структура данных непрозрачна для Storage — интерпретация выполняется потребителем через Reader.

Смещение данных от начала фблока: `32 + params_size + catalog_size + 12`.

## 9. TOC (Table of Contents)

TOC — оглавление данных внутри фблока. Расположен после данных, перед эпилогом. Для Storage — непрозрачный массив байтов. Парсинг и интерпретация TOC — ответственность Reader. При реализации Reader может использовать формат узлов дерева (раздел 4) для структурирования TOC, но это не является требованием Storage.

Размер TOC определяется полем `toc_size` из эпилога.

Смещение TOC от конца фблока: `12 + toc_size` (отсчёт от конца).

## 10. Эпилог

Эпилог — фиксированная секция в конце фблока. Читается с конца фблока (последние 12 байт).

| Смещение от конца | Размер | Тип    | Поле           | Описание |
|-------------------|--------|--------|----------------|----------|
| -12               | 4      | uint32 | crc32_data     | CRC32 данных (payload). |
| -8                | 4      | uint32 | crc32_toc      | CRC32 TOC. |
| -4                | 4      | uint32 | toc_size       | Размер TOC в байтах. |

**Итого: 12 байт.**

### 10.1. Роль эпилога при восстановлении после сбоя

Эпилог записывается последним. При сбое питания во время записи фблока (состояние `in_progress`) эпилог позволяет определить степень завершённости записи:

| crc32_data | crc32_toc | Интерпретация |
|------------|-----------|---------------|
| ✓          | ✓         | Запись завершена полностью. Данные и TOC целы. |
| ✓          | ✗         | Данные записаны, TOC — частично. |
| ✗          | —         | Данные записаны частично. |

## 11. Алгоритм сканирования при старте

При старте Storage необходимо найти фблок с самым свежим каталогом.

1. Для каждого фблока прочитать 8 байт со смещением 4 от начала (`catalog_time`).
2. Выбрать фблок с максимальным `catalog_time`.
3. Прочитать полный пролог выбранного фблока (32 байта).
4. Проверить магическое число.
5. Прочитать параметры и каталог.
6. Проверить контрольные суммы (crc32_fixed, crc32_params, crc32_catalog).
7. Если проверка не прошла — перейти к фблоку со следующим по свежести `catalog_time`.

Суммарный объём чтения на этапе сканирования: `N × 8` байт, где N — количество фблоков.

## 12. Карта смещений

Сводная таблица смещений всех секций от начала фблока:

| Секция | Начало | Размер |
|--------|--------|--------|
| Пролог (фиксированная часть) | 0 | 32 |
| Параметры | 32 | params_size |
| Каталог | 32 + params_size | catalog_size |
| Контрольные суммы заголовка | 32 + params_size + catalog_size | 12 |
| Данные | 32 + params_size + catalog_size + 12 | вычисляется |
| TOC | fblock_size - 12 - toc_size | toc_size |
| Эпилог | fblock_size - 12 | 12 |

Размер данных: `fblock_size - 32 - params_size - catalog_size - 12 - toc_size - 12`.
