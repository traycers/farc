# Бинарный формат фблока

## 1. Назначение документа

Документ описывает бинарный формат фблока: расположение и размеры полей, порядок байтов, контрольные суммы, правила парсинга. Является спецификацией для реализации чтения и записи фблоков.

Документ **не описывает**:
- Концептуальную модель данных, компоненты и их ответственность — см. `01-storage.md`.
- Алгоритмы операций (сканирование при старте, выбор фблока для записи, расширение, сужение) — см. `03-storage-operations.md`.

## 2. Общие соглашения

- Порядок байтов: **Little Endian** (LE) для всех многобайтовых целых.
- Временные метки: **Unix timestamp в наносекундах**, uint64.
- Контрольные суммы: **CRC32** (IEEE 802.3, полином `0x04C11DB7`).
- Строки: UTF-8, без BOM.

## 3. Общая структура фблока

Фблок — непрерывная область фиксированного размера на диске. Состоит из шести секций:

```
┌─────────────────────────────────────────┐
│ Пролог                                  │  Фиксированная часть + параметры (JSON)
├─────────────────────────────────────────┤
│ Каталог                                 │  Снимок состояния всех фблоков (SoA)
├─────────────────────────────────────────┤
│ Контрольные суммы заголовка             │  CRC32 × 3
├─────────────────────────────────────────┤
│ Данные                                  │  Payload (фчанки)
├─────────────────────────────────────────┤
│ TOC                                     │  Оглавление данных (непрозрачно для Storage)
├─────────────────────────────────────────┤
│ Эпилог                                  │  magic + CRC32 данных + CRC32 TOC + размер TOC
└─────────────────────────────────────────┘
```

Чтение фблока выполняется с двух сторон: пролог и каталог — с начала, эпилог — с конца.

## 4. Версионирование формата

Формат имеет мажорную и минорную версии, записанные в фиксированной части пролога:

- **Мажорная версия** (`format_version_major`, uint16) — изменяется при несовместимых изменениях. Библиотека отказывается работать с хранилищем несовместимой мажорной версии.
- **Минорная версия** (`format_version_minor`, uint16) — изменяется при обратно совместимых расширениях. Библиотека старшей минорной версии читает хранилища младших минорных версий той же мажорной версии.

Текущая версия формата: **v1.0**.

## 5. Пролог

Пролог — начальная секция фблока. Содержит фиксированную часть и параметры хранилища в формате JSON.

### 5.1. Фиксированная часть

Фиксированная часть пролога **никогда не меняет свою структуру**. Поля, их размеры и смещения постоянны для всех версий формата — это гарантия того, что любая библиотека может прочитать базовую идентификацию фблока.

| Смещение | Размер | Тип    | Поле                   | Описание |
|----------|--------|--------|------------------------|----------|
| 0        | 8      | bytes  | magic_prolog           | `0x46 0x41 0x52 0x43 0x50 0x52 0x4F 0x4C` (`FARCPROL`). Идентифицирует начало фблока. |
| 8        | 2      | uint16 | format_version_major   | Мажорная версия формата. |
| 10       | 2      | uint16 | format_version_minor   | Минорная версия формата. |
| 12       | 4      | —      | reserved               | Зарезервировано, должно быть 0. |
| 16       | 8      | uint64 | write_sequence         | Монотонный счётчик записей (ADR-008). Определяет порядок фблоков. |
| 24       | 8      | uint64 | catalog_time           | Время создания снимка каталога в Unix timestamp наносекундах. Для аудита и диагностики. |
| 32       | 8      | uint64 | fblock_size            | Размер фблока в байтах. |
| 40       | 4      | uint32 | params_size            | Размер секции параметров в байтах. |
| 44       | 4      | uint32 | catalog_entry_count    | Количество записей в каталоге (равно количеству фблоков в Storage). |
| 48       | 4      | uint32 | catalog_size           | Размер каталога в байтах. |
| 52       | 4      | —      | reserved               | Зарезервировано, должно быть 0. |

**Итого: 56 байт.**

Отсутствие валидного `magic_prolog` означает, что фблок находится в состоянии `uninitialized` (ADR-006) — заголовок никогда не записывался.

### 5.2. Параметры (JSON)

Параметры хранилища следуют сразу за фиксированной частью, размер определяется полем `params_size`. Сериализованы в JSON (UTF-8, без BOM) согласно ADR-009.

Правила эволюции:
- Новые параметры — новые ключи в JSON.
- Парсер игнорирует незнакомые ключи (forward compatibility).
- Обязательные параметры, отсутствующие в JSON, — ошибка формата.

Ожидаемые параметры:

| Ключ | Тип | Описание |
|------|-----|----------|
| `fchunk_size` | integer | Размер фчанка в байтах. |
| `read_chunk_size` | integer | Размер порции чтения в байтах (ADR-005). По умолчанию равен `fchunk_size`. Должен быть кратен размеру кластера диска. |
| `recording.mode` | string | Режим записи: `"cyclic"` или `"fill_once"`. |
| `retention.days` | integer | Минимальный срок хранения фблока в днях после `end` (последнего timestamp данных в фблоке). |

Пример:

```json
{
  "fchunk_size": 4194304,
  "recording": {
    "mode": "cyclic"
  },
  "retention": {
    "days": 30
  }
}
```

### 5.3. Смещение следующей секции

Начало каталога: `56 + params_size`.

## 6. Каталог

Каталог — снимок состояния всех фблоков Storage на момент начала записи текущего фблока. Расположен сразу после пролога. Сериализован в виде **структуры массивов** (Structure of Arrays, SoA): для каждого поля — отдельный плотно упакованный массив длины `catalog_entry_count`.

### 6.1. Компоновка

Массивы располагаются последовательно, без заголовков между ними. Порядок и формат фиксированы мажорной версией формата.

Для версии v1.0:

| Смещение от начала каталога | Размер (байт) | Поле     | Элемент |
|-----------------------------|---------------|----------|---------|
| 0                           | N × 1         | flags[]  | uint8 — состояние и флаги (см. 6.2) |
| N × 1                       | N × 16        | uuid[]   | 16 байт — UUID фблока |
| N × 17                      | N × 8         | begin[]  | uint64 — время начала актуальности (ns) |
| N × 25                      | N × 8         | end[]    | uint64 — время окончания актуальности (ns) |

где `N` = `catalog_entry_count`.

**Итого на запись: 33 байта.**

**Полный размер каталога:** `catalog_size = N × 33`.

Для 1 миллиона фблоков каталог занимает ≈ 33 МБ.

Порядок массивов (flags первым) выбран сознательно: при сканировании можно прочитать только массив flags (1 байт на фблок) и быстро получить карту состояний без чтения UUID и временных меток.

### 6.2. Кодирование flags

Байт flags:

| Бит | Маска | Значение | Описание |
|-----|-------|----------|----------|
| 0-1 | 0x03  | 0        | `uninitialized` |
|     |       | 1        | `in_progress` |
|     |       | 2        | `ready` |
|     |       | 3        | `bad` |
| 2   | 0x04  | 1        | `protected` |
| 3-7 |       |          | зарезервированы, должны быть 0 |

Значение `uninitialized = 0` соответствует нулевому байту — естественное состояние свежевыделенной или нулевой области памяти/диска (ADR-006).

Для `uninitialized` фблоков значения в массивах `uuid[]`, `begin[]`, `end[]` должны быть нулевыми (не используются, но должны быть предсказуемыми для CRC-проверки).

### 6.3. Свойство снимка

Снимок каталога в заголовке фблока `K` отражает состояние Storage на момент начала записи `K`:
- Сам `K` в собственном снимке помечен `in_progress`.
- Предыдущий успешно записанный фблок помечен `ready`.

Подтверждение «фблок `M` успешно записан» попадает на диск в каталоге следующего записанного фблока `M+1`.

## 7. Контрольные суммы заголовка

Три контрольных суммы расположены последовательно сразу после каталога. Позволяют определить, какая именно часть заголовка повреждена.

| Смещение от начала секции | Размер | Тип    | Поле            | Покрывает |
|---------------------------|--------|--------|-----------------|-----------|
| 0                         | 4      | uint32 | crc32_fixed     | Фиксированная часть пролога (байты 0..55). |
| 4                         | 4      | uint32 | crc32_params    | JSON параметров целиком. |
| 8                         | 4      | uint32 | crc32_catalog   | Каталог целиком. |

**Итого: 12 байт.**

Смещение секции от начала фблока: `56 + params_size + catalog_size`.

### 7.1. Диагностика повреждений

| crc32_fixed | crc32_params | crc32_catalog | Интерпретация |
|-------------|--------------|---------------|---------------|
| ✓           | ✓            | ✓             | Заголовок целый. |
| ✓           | ✓            | ✗             | Параметры доступны, снимок каталога потерян. |
| ✓           | ✗            | ✓             | Каталог доступен, параметры повреждены. |
| ✓           | ✗            | ✗             | Только фиксированная часть валидна. |
| ✗           | —            | —             | Фблок нечитаем — либо повреждён, либо `uninitialized`. Для различения смотрят на `magic_prolog`: совпадает → повреждён, не совпадает → `uninitialized`. |

## 8. Данные

Данные (payload) расположены после контрольных сумм заголовка. Записываются и верифицируются пофчанково (write-verify). Внутренняя структура данных непрозрачна для Storage — интерпретация выполняется потребителем через Reader.

Смещение данных от начала фблока: `56 + params_size + catalog_size + 12`.

## 9. TOC (Table of Contents)

TOC — оглавление данных внутри фблока. Расположен после данных, перед эпилогом. Для Storage — непрозрачный массив байтов. Парсинг и интерпретация TOC — ответственность Reader. Формат TOC будет описан в отдельном документе при проектировании слоя полезных данных фблока.

Размер TOC определяется полем `toc_size` из эпилога.

Смещение TOC от конца фблока: `20 + toc_size`.

## 10. Эпилог

Эпилог — фиксированная секция в конце фблока. Читается с конца фблока (последние 20 байт).

| Смещение от конца | Размер | Тип    | Поле           | Описание |
|-------------------|--------|--------|----------------|----------|
| -20               | 8      | bytes  | magic_epilog   | `0x46 0x41 0x52 0x43 0x45 0x50 0x49 0x4C` (`FARCEPIL`). Отличается от `magic_prolog` — позволяет распознать эпилог даже при случайном чтении. |
| -12               | 4      | uint32 | crc32_data     | CRC32 данных (payload). |
| -8                | 4      | uint32 | crc32_toc      | CRC32 TOC. |
| -4                | 4      | uint32 | toc_size       | Размер TOC в байтах. |

**Итого: 20 байт.**

### 10.1. Роль эпилога при восстановлении после сбоя

Эпилог записывается последним. Наличие валидного `magic_epilog` — признак того, что запись фблока дошла до финальной стадии. При сбое питания во время записи фблока эпилог позволяет определить степень завершённости записи:

| magic_epilog | crc32_data | crc32_toc | Интерпретация |
|--------------|------------|-----------|---------------|
| ✓            | ✓          | ✓         | Запись завершена полностью. Данные и TOC целы. |
| ✓            | ✓          | ✗         | Эпилог записан, но TOC повреждён (крайне редко — записывается до эпилога). |
| ✓            | ✗          | —         | Эпилог записан, данные повреждены (физический дефект). |
| ✗            | —          | —         | Запись не дошла до эпилога — фблок `in_progress` на момент сбоя. |

## 11. Карта смещений

Сводная таблица смещений всех секций от начала фблока:

| Секция | Начало | Размер |
|--------|--------|--------|
| Пролог (фиксированная часть) | 0 | 56 |
| Параметры (JSON) | 56 | params_size |
| Каталог | 56 + params_size | catalog_size = N × 33 |
| Контрольные суммы заголовка | 56 + params_size + catalog_size | 12 |
| Данные | 56 + params_size + catalog_size + 12 | вычисляется |
| TOC | fblock_size - 20 - toc_size | toc_size |
| Эпилог | fblock_size - 20 | 20 |

Размер данных: `fblock_size - 56 - params_size - catalog_size - 12 - toc_size - 20 = fblock_size - 88 - params_size - catalog_size - toc_size`.
