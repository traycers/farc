# Требования к системе хранения данных

## 1. Бизнес-контекст
Система предназначена для управления данными, организованными в виде **фблоков (функциональных блоков)**. Каждый фблок — это самодостаточная единица данных, которая может храниться на диске, а также (вне нашей ответственности) передаваться по сети или обрабатываться независимо от других фблоков.

**Ключевая идея:** фблок — это логическая единица работы приложения. То, с чем оперирует пользователь: он запрашивает запись фблока, чтение фблока. Передача по сети — ответственность потребителя.

## 2. Терминология

| Термин | Описание |
|--------|----------|
| **Фблок (fblock)** | Функциональный блок — логическая единица данных, с которой работает приложение. Имеет фиксированный размер (например, 512 МБ, 2 ГБ). Содержимое фблока самодостаточно и не зависит от других фблоков. |
| **Фчанк (fchunk)** | Часть фблока, минимальная единица для операций ввода-вывода на диске и проверки записи. Один фблок состоит из множества фчанков. |
| **Битый фчанк** | Фчанк, в который невозможно надёжно записать данные (физический дефект диска). |
| **Битый фблок** | Фблок, в котором обнаружен хотя бы один битый фчанк при записи. Такой фблок считается непригодным для хранения данных. |
| **Защищённый фблок** | Фблок, помеченный пользователем как read-only (запрет на запись). |
| **Потребитель** | Процесс (например, "архив"), который использует библиотеку для работы с фблоками. Потребитель: <br>• Заполняет фблоки данными и отдаёт их на запись.<br>• Запрашивает чтение фблоков (целиком или частично).<br>• Работает с метриками, алертами и логами.<br>• Может управлять несколькими экземплярами библиотеки (каждый со своим **хранилищем**). |
| **Хранилище (storage)** | Общий термин для места, где хранятся данные. Может быть:<br>• **Файл** (Windows всегда, Linux опционально).<br>• **Раздел/блочное устройство** (Linux, для максимальной производительности).<br>Способ работы задаётся при инициализации. |
| **Время начала актуальности** | Момент времени, с которого данные во фблоке считаются актуальными. **Задаётся потребителем при заполнении фблока (Unix nanoseconds).** |
| **Время окончания актуальности** | Момент времени, после которого данные во фблоке считаются неактуальными. **Задаётся потребителем при заполнении фблока (Unix nanoseconds). Является основой для принятия решения о перезаписи.** |
| **Время записи** | **Момент начала записи фблока на диск. Заполняется библиотекой, используется для быстрого старта.** |

### Важно о терминах
- **Сейчас:** в текущей реализации на диске фблок целиком соответствует непрерывной области на диске. Фчанки нужны только для внутренних операций (проверка записи, обработка сбоев).
- **В будущем:** фблок может храниться распределённо, передаваться по сети, кэшироваться в памяти. Но для приложения он остаётся неделимой единицей с фиксированным размером.

## 3. Требования к фблоку как самодостаточной единице

### 3.1. Независимость
- Фблок должен содержать всю информацию, необходимую для интерпретации своих данных.
- Для чтения и использования фблока **не требуется** обращаться к метаданным других фблоков или глобальному заголовку раздела (после того как фблок извлечён из системы хранения).

### 3.2. Идентификация

В системе используются два разных идентификатора с разными целями:

| Идентификатор | Назначение | Где хранится | Кто использует |
|---------------|------------|--------------|----------------|
| **Индекс фблока** | Физический адрес на диске для циклической записи. Порядковый номер фблока от 0 до N-1, где N — общее количество фблоков в разделе. | Только в метаданных раздела (таблица состояния). | Внутренняя реализация записи. |
| **UUID фблока** | Логический идентификатор, уникальный навсегда. Не зависит от физического расположения. | **В двух местах:**<br>1. В заголовке самого фблока на диске.<br>2. В метаданных раздела (маппинг UUID → индекс). | Потребитель данных (клиент). |

#### Почему два идентификатора?
- **Индекс** нужен для эффективной циклической записи — быстро перебирать фблоки по порядку.
- **UUID** нужен для клиента — чтобы данные можно было найти даже после перезаписи старого физического места.

#### Важное требование
UUID **обязательно хранится в самом фблоке**. Это позволяет:
1. Проверять целостность при чтении (совпадает ли UUID в метаданных с тем, что реально лежит в фблоке).
2. Восстанавливать метаданные в случае их повреждения — сканированием диска и чтением заголовков фблоков.

#### Формат UUID
- Используется **UUIDv4** (128 бит, 16 байт).
- **Генерируется библиотекой случайным образом после успешной записи фблока на диск.**
- Потребитель получает UUID только через механизмы уведомления (см. раздел 4.12).
- Отлично поддерживается PostgreSQL (тип `uuid`, индексы).

### 3.3. Состав фблока

Фблок состоит из двух частей:
1. **Заголовок библиотеки** — служебная информация, добавляемая системой хранения.
2. **Полезные данные** — данные, переданные потребителем.


### 3.4. Инварианты
- Размер фблока (включая служебную информацию) **строго фиксирован** для всего хранилища.
- Размер фчанка **строго фиксирован** для хранилища.
- **Заголовок всегда находится в начале фблока.**
- При передаче по сети фблок передаётся целиком (заголовок + данные).
- Получатель фблока должен иметь возможность проверить его целостность и определить формат данных внутри.

## 4. Функциональные требования

### 4.1. Инициализация раздела

Процесс подготовки раздела к работе состоит из двух этапов:

#### 4.1.1. Резервирование места (только для файлов)
Если библиотека работает с обычным файлом (на Windows или Linux), она должна уметь создать файл нужного размера и зарезервировать под него место на диске, чтобы другие процессы не могли использовать это пространство.

**Требования:**
- Поддержка создания файла указанного размера (например, 100 ГБ, 20 ТБ).
- Возможность работать с уже существующим файлом (пропуск резервирования).
- При создании файла использовать эффективные механизмы (sparse files или выделение всего места — настраивается).

#### 4.1.2. Инициализация структуры раздела
Библиотека должна уметь подготовить раздел к работе: записать служебную информацию, необходимую для дальнейшего функционирования.

**Что делает инициализация:**
- Записывает глобальный заголовок раздела (магическое число, версию формата).
- Сохраняет параметры: размер фблока, размер фчанка, **глобальный срок хранения по умолчанию**.
- Инициализирует таблицы метаданных (битые фблоки, защищённые фблоки, маппинг UUID->индекс, указатель записи).
- Создаёт резервные копии критических структур (если требуется).

**Важно:** Инициализация не должна трогать данные за пределами служебных областей.

#### 4.1.3. Идемпотентность
- Повторная инициализация уже работающего раздела должна быть либо запрещена, либо требовать явного подтверждения (флаг `--force`), так как уничтожает существующие метаданные.

#### 4.1.4. Готовность к работе
После успешной инициализации раздел должен быть готов к использованию:
- Библиотека может его открыть.
- Все структуры метаданных валидны.
- Можно начинать циклическую запись.

### 4.2. Управление разделами
- Система должна работать с дисковыми разделами как с хранилищем фблоков.
- Раздел содержит последовательность фблоков (физически — непрерывную область, логически — массив фблоков).
- Раздел идентифицируется путём к устройству или файлу.

### 4.3. Организация хранения на диске
- При хранении на диске фблок **физически занимает непрерывную область** (сейчас) или может быть фрагментирован (в будущем).
- Для эффективной работы с диском и обработки сбоев фблок делится на **фчанки**.
- Размер фчанка задаётся при инициализации раздела (рекомендуется 4–16 МБ).

**Связь фблоков и фчанков:**
- Фблок размером N состоит из K фчанков, где K = N / размер_фчанка (результат должен быть целым).
- Операции записи/чтения с диска всегда идут по фчанкам.
- Проверка успешности записи тоже идёт по фчанкам.

### 4.4. Операции с фблоками

#### 4.4.1. Запись фблока

1. **Выбор следующего доступного фблока:**
   Система движется по кругу, начиная с указателя последней записи, и ищет первый фблок, который:
   - Не битый.
   - Не защищённый пользователем.
   - Либо свободен, либо занят **и время окончания актуальности ≤ текущее время**.

   **Порядок проверки при обходе:**
   1. Пропустить битые.
   2. Пропустить защищённые пользователем.
   3. Если фблок занят — проверить **время окончания актуальности**:
      - Если ещё не наступило — пропустить (данные ещё актуальны).
      - Если наступило или прошло — можно использовать для перезаписи.

   ***Важно:** Время записи не влияет на возможность перезаписи.*

2. Если после полного оборота подходящий фблок не найден:
   - Система генерирует аварию "нет свободного места для записи".
   - Требуется вмешательство администратора (очистка, добавление места, принудительное удаление защищённых).

3. Для каждого фчанка внутри выбранного фблока:
   - Записать фчанк на диск.
   - Прочитать записанный фчанк обратно.
   - Сравнить с оригиналом в памяти.
   - При несовпадении:
     - Повторить попытку записи для данного фчанка (до N раз, N из конфига, по умолчанию 3).
     - Если все попытки исчерпаны — **считать весь фблок битым** и немедленно перейти к шагу 5.

4. Если все фчанки записаны успешно:
   - Сгенерировать UUID для фблока.
   - Заполнить поле UUID и время записи в заголовке фблока.
   - Сменить статус фблока на "занят".
   - Обновить метаданные (маппинг UUID->индекс).
   - Фблок считается успешно записанным.
   - Отправить уведомление о новом фблоке (см. 4.12).
   - Конец.

5. **Обработка битого фблока:**
   - Пометить текущий фблок как **битый** в метаданных раздела.
   - Перейти к следующему фблоку по кругу (пропуская защищённые и уже битые).
   - Начать запись **того же самого фблока данных** заново с шага 1 (на новом месте).

**Важно:** Фблок данных (логическая единица) либо сохраняется целиком в некотором физическом фблоке, либо не сохраняется вообще. Частичная запись недопустима.

#### 4.4.2. Чтение фблока
1. Найти фблок по идентификатору (UUID) через маппинг в метаданных.
2. Определить его физическое положение на диске (индекс и смещение).
3. Прочитать запрошенные данные с диска.

**Поддержка частичного чтения:**
- Система должна позволять читать любой непрерывный диапазон внутри фблока (offset, length).
- Чтение части фблока **не должно** требовать чтения всего фблока.
- Размер чтения не привязан к размеру фчанка — можно читать 1 байт, 48 МБ, 2 ГБ.
- Фчанки — это деталь реализации записи, они не видны при чтении.

#### 4.4.3. Приоритеты операций
- Запись имеет абсолютный приоритет над чтением.
- Если одновременно приходят команды на запись и чтение, запись выполняется первой.
- Чтение выполняется только в паузах между записями.

### 4.5. Управление состоянием

| Состояние фблока | Описание |
|------------------|----------|
| **Свободен** | Фблок не содержит данных, доступен для записи |
| **Занят (актуален)** | Фблок содержит данные, и **время окончания актуальности** ещё не наступило. **Перезапись запрещена**. |
| **Занят (неактуален)** | Фблок содержит данные, но **время окончания актуальности** уже наступило. **Может быть перезаписан** при циклической записи. |
| **Защищён пользователем** | Фблок помечен пользователем как read-only. Перезапись запрещена независимо от времени. |
| **Битый** | Фблок повреждён физически, запись/чтение невозможны |

### 4.6. Обработка битых фчанков
- **Важное упрощение:** битые фчанки не обрабатываются индивидуально на уровне фблока.
- Если при записи обнаруживается битый фчанк — весь фблок считается битым.
- Информация о битых фчанках всё равно может храниться в метаданных раздела для диагностики и статистики, но не влияет на логику записи фблоков.
- При циклической записи битые фблоки просто пропускаются.

### 4.7. Защита от записи (write protection)
- Библиотека предоставляет функцию/метод для пометки фблока как защищённого.
- Параметры:
  - Идентификатор фблока (UUID или индекс).
  - Опционально: причина защиты (для логирования).
- Защита сохраняется в метаданных раздела и переживает перезапуск.
- Защищённый фблок пропускается при циклической записи.
- Попытка записи в защищённый фблок возвращает ошибку.
- Снятие защиты — отдельная функция.

### 4.8. Очередь записи и конкурентное наполнение

#### 4.8.1. Принцип работы
- В памяти поддерживается **пул буферов фблоков** (количество настраивается в конфиге).
- Один фблок записывается на диск, другой (или несколько) наполняется данными из сети.
- Готовые к записи фблоки ставятся в **очередь на запись** (FIFO).
- Очередь существует **только в памяти**, не сбрасывается на диск.
- **Важно: UUID для фблока генерируется только после успешной записи на диск. Фблоки в очереди ещё не имеют UUID и не видны потребителю.**

#### 4.8.2. Контроль переполнения (backpressure)

Если скорость наполнения фблоков превышает скорость их записи на диск, очередь растёт. При достижении предельного размера очереди система применяет **обратное давление (backpressure)**:

- Система сигнализирует потребителю о состоянии перегрузки при запросе нового буфера для заполнения.
- Запись на диск продолжается в штатном режиме.
- Как только очередь уменьшается до безопасного уровня, сигнал о перегрузке снимается.

**Важно:**
- Библиотека **не анализирует** содержимое фблоков и не принимает решений о том, какие данные важнее.
- Библиотека **не отбрасывает** фблоки — она только сигнализирует потребителю о переполнении через механизм обратной связи при запросе буфера.
- Потребитель сам решает, что делать, получив сигнал о перегрузке:
  - Отбросить менее важные данные.
  - Увеличить время заполнения фблока (чтобы дать системе больше времени на запись).
  - Отправить алерт администратору.
  - И т.д.

**Принцип:** библиотека отвечает за запись и целостность, потребитель — за стратегию работы с данными при перегрузках.

#### 4.8.3. Метрики очереди
Для контроля состояния очереди собираются метрики:
- Текущий размер очереди.
- Максимальный размер очереди (пики).
- Скорость наполнения (байт/сек из сети).
- Скорость записи (байт/сек на диск).
- Количество срабатываний backpressure.
- Время ожидания фблока в очереди.

#### 4.8.4. Двухфазный протокол записи

Взаимодействие с потребителем строится по двухфазному протоколу:

1.  **Фаза получения буфера:**
    - Потребитель запрашивает у библиотеки буфер для заполнения.
    - Библиотека предоставляет буфер фиксированного размера (равного размеру фблока минус размер заголовка).
    - Ответ на запрос может быть:
        *   *Успех:* буфер предоставлен.
        *   *Предупреждение:* буфер предоставлен, но очередь близка к переполнению (порог настраивается).
        *   *Перегрузка:* буфер предоставлен, но система в состоянии backpressure (порог настраивается). Потребитель должен скорректировать свою стратегию.
        *   *Ошибка "нет свободных буферов":* все буферы заняты. Потребитель должен повторить попытку позже.

2.  **Фаза передачи заполненного буфера:**
    - Потребитель заполняет буфер данными и передаёт его обратно библиотеке вместе с метаданными:
        *   Время начала актуальности.
        *   Время окончания актуальности.
    - Библиотека ставит буфер в очередь на запись.
    - Потребитель может вернуть буфер без данных (освободить), если передумал сохранять.

### 4.9. Мониторинг состояния диска

Система должна отслеживать количество битых фблоков и бить тревогу при превышении порога.

**Требования:**
- **Пороговое значение:** 5% битых фблоков от общего количества (настраивается в конфиге).
- При превышении порога система должна:
  - Сгенерировать алерт (в лог, через API, в внешнюю систему мониторинга).
  - Продолжить работу (запись не останавливается).
- Администратор должен быть уведомлен о необходимости замены диска.

### 4.10. Консистентность при сбоях

#### 4.10.1. Проблема
При сбое питания в момент записи фблока возможна ситуация, когда часть фчанков уже записана на диск, а часть — нет. Фблок оказывается в **незавершённом состоянии**.

#### 4.10.2. Требования к обнаружению незавершённых фблоков

Система должна иметь механизм для определения фблоков, запись которых не была завершена.

В заголовке каждого фблока предусмотрено поле **статус записи** (1 байт), которое может принимать значения:
- `0x00` — фблок свободен (не содержит данных)
- `0x01` — фблок занят (корректно записан)
- `0x02` — фблок в процессе записи (незавершённый)
- `0x03` — фблок битый

**Алгоритм при записи:**
1. Перед началом записи фблока выставить статус `в процессе записи`.
2. Записать все фчанки.
3. После успешной записи всех фчанков сменить статус на `занят` и заполнить UUID.

#### 4.10.3. Поведение при старте системы
При запуске после сбоя система должна:
1. Просканировать метаданные (или заголовки фблоков) на предмет статуса `в процессе записи`.
2. Все фблоки с таким статусом **считать битыми** и пометить их в таблице битых фблоков.
3. Освободить связанные с ними записи в маппинге UUID → индекс (если были).
4. Продолжить нормальную работу.

**Важно:** система не пытается дописать незавершённые фблоки — это задача отдельной CLI программы восстановления.

#### 4.10.4. Восстановление незавершённых фблоков
CLI программа восстановления (раздел 8) должна уметь:
- Находить фблоки со статусом `в процессе записи`.
- Анализировать, какие данные удалось записать.
- Извлекать сохранившиеся данные (если это возможно и нужно).
- По желанию пользователя — пытаться восстановить фблок или просто удалять его из метаданных.

#### 4.10.5. Журналирование (WAL) для метаданных
Для атомарности обновления метаданных (таблицы битых фблоков, маппинг UUID → индекс) используется журналирование (WAL). Это гарантирует, что:
- При сбое во время обновления метаданных система сможет восстановить консистентное состояние.
- Не будет ситуации, когда фблок помечен как занятый, но UUID отсутствует в маппинге (или наоборот).

### 4.11. Уведомление об удалении фблока

#### 4.11.1. Проблема
При циклической записи система перезаписывает старые фблоки новыми данными. Физический фблок с индексом N, который раньше содержал данные с UUID=X, теперь содержит данные с UUID=Y.

Потребитель, который хранит UUID=X, не знает, что его данные больше не существуют.

#### 4.11.2. Требование
Система должна уведомлять потребителя о том, что фблок с определённым UUID **удалён** (перезаписан).

#### 4.11.3. Механизм уведомления
Библиотека должна предоставлять механизм, через который приложение может получать уведомления о событиях удаления фблоков.

**Событие удаления должно содержать:**
- UUID удалённого фблока.
- Время удаления (timestamp).
- Причину удаления (например, "циклическая перезапись").

#### 4.11.4. Когда происходит уведомление
Уведомление отправляется в момент, когда система приняла решение перезаписать фблок:
1. Система выбрала индекс для новой записи.
2. Проверила, что фблок по этому индексу currently занят (имеет валидный UUID).
3. **Перед** фактической перезаписью данных отправляет уведомление о том, что старый UUID будет удалён.
4. После уведомления выполняет запись новых данных.

#### 4.11.5. Гарантии доставки
- Уведомления отправляются **до** физической перезаписи, чтобы потребитель мог отреагировать.
- Система не ждёт подтверждения от потребителя (асинхронное уведомление).
- При сбое питания между уведомлением и записью возможна ситуация, когда уведомление было отправлено, но данные не перезаписались. Это допустимо — потребитель получил "ложное" уведомление, но при следующем старте фблок всё ещё будет содержать старые данные. В этом случае UUID остаётся валидным.

#### 4.11.6. Интеграция с мониторингом
События удаления должны также:
- Логироваться (для аудита).
- Учитываться в метриках (количество удалённых фблоков).
- Быть доступны через API статистики.

### 4.12. Уведомление о новом фблоке

#### 4.12.1. Проблема
Потребитель заполняет фблок данными и отдаёт его библиотеке на запись. **Библиотека генерирует UUID для этого фблока только после успешной записи на диск.**

Потребитель должен узнать, какой UUID получил фблок после успешной записи.

#### 4.12.2. Как потребитель получает UUID

| Способ | Описание |
|--------|----------|
| **Синхронное ожидание** | **Функция передачи заполненного буфера может быть вызвана в синхронном режиме. В этом случае она блокируется до завершения записи фблока на диск и возвращает UUID.** |
| **Асинхронное уведомление** | Библиотека отправляет событие о том, что новый фблок успешно записан. Потребитель должен быть подписан на такие события. |

#### 4.12.3. Требование
Библиотека должна предоставлять **оба механизма**:
- **Синхронный вызов, возвращающий UUID.**
- Асинхронное уведомление о новой записи (для случаев, когда потребитель не хочет блокироваться).

#### 4.12.4. Событие о новом фблоке (асинхронный вариант)

**Событие должно содержать:**
- UUID нового фблока.
- Время записи (timestamp).
- Индекс фблока (опционально, для диагностики).
- Размер полезных данных (если отличается от размера фблока).

#### 4.12.5. Когда происходит уведомление
Уведомление отправляется **после** успешного завершения записи фблока:
1. Все фчанки записаны и проверены.
2. Метаданные обновлены (маппинг UUID → индекс).
3. Фблок помечен как "занят".
4. Отправляется уведомление потребителю.

#### 4.12.6. Интеграция с мониторингом
- Логируются все события о новых фблоках.
- Метрика: количество записанных фблоков.
- **Метрика: время между фазой получения буфера и завершением записи.**

## 5. Требования к самодостаточности фблока (подробно)

### 5.1. Что должен содержать фблок (на уровне требований)
- **Идентификацию формата** (магическое число, версию) — чтобы получатель мог определить, как интерпретировать данные.
- **Уникальный идентификатор** фблока (UUID).
- **Временные метки актуальности**.
- **Контрольную информацию** для проверки целостности (контрольные суммы заголовка и данных).
- **Полезные данные**.

### 5.2. Чего НЕ должен требовать фблок
- Глобального заголовка раздела для интерпретации данных.
- Информации о соседних фблоках.
- Внешних таблиц (битых фблоков, защищённых фблоков) — это метаданные **хранилища**, а не фблока.

### 5.3. Следствия для архитектуры
- Метаданные раздела (битовые карты, таблицы) хранятся отдельно от фблоков.
- При передаче фблока по сети метаданные раздела **не передаются**.
- Получатель должен иметь возможность проверить фблок, даже если у него нет контекста раздела.

## 6. Нефункциональные требования

### 6.1. Масштабируемость
- Разделы до 20 ТБ и более.
- Количество фблоков: до нескольких миллионов (при размере фблока 1 ГБ — 20 000 фблоков на 20 ТБ).
- Количество фчанков: до сотен миллионов (при фчанке 4 МБ — 5 млн фчанков на 20 ТБ).
- 64-битная адресация всего.

### 6.2. Производительность
- Запись с проверкой (write-verify) должна упираться в скорость диска.
- При обнаружении битого фблока — немедленный переход к следующему, минимум задержки.
- Поддержка частичного чтения без лишних операций ввода-вывода.
- Минимизация накладных расходов на metadata.

### 6.3. Надёжность
- Контрольные суммы для всех критических данных (заголовки фблоков, таблицы состояния).
- Атомарность обновления метаданных (журналирование WAL).
- При обнаружении битого фчанка — немедленное обновление метаданных (пометка фблока как битого).
- Минимум 2 копии критических метаданных (заголовок раздела) в разных зонах диска.

### 6.4. Переносимость
- Фблок, созданный на Linux, должен корректно читаться на Windows (как набор байт).
- Все числа в заголовках фблоков — **little-endian** (единый порядок для всех платформ).
- Формат не должен зависеть от особенностей конкретной файловой системы.

- **Версионность:** хранить версию формата в заголовке каждого фблока (1 байт).
- **Избыточность:** хранить контрольную сумму заголовка и данных.
- **Защита заголовка:** дублировать критические поля (по возможности).

### 7.6. Поведение при чтении битого фблока
- **Всегда ошибка.**
- Восстановление — отдельный процесс (утилита), не на лету.

### 7.7. Циклическая запись с пропуском битых/защищённых
- Хранить указатель "последний успешно записанный фблок".
- При выборе следующего пропускать битые, защищённые и те, у которых **время окончания актуальности ещё не наступило**.
- Если все фблоки битые/защищённые/актуальные — авария, требуется вмешательство.

### 7.8. Атомарность операций
- **Требуется журналирование (WAL)** для метаданных.
- При сбое питания в середине записи фблока:
  - Фблок считается невалидным (статус "в процессе записи").
  - При старте проверять журнал и незавершённые фблоки, помечать их как битые.

### 7.9. Мониторинг и статистика

#### 7.9.1. Принцип сбора метрик
Библиотека должна собирать и предоставлять метрики работы, но **не хранить их историю** на диске. Хранение и визуализация — задачи внешних систем (Prometheus, Grafana, Loki).

#### 7.9.2. Какие метрики собирать

**Метрики записи:**
- Время записи каждого фблока (миллисекунды)
- Количество попыток записи на фблок
- Количество битых фчанков при записи
- Скорость записи (байт/сек) — мгновенная и средняя
- Интервалы между записями фблоков

**Метрики чтения:**
- Частота чтения по фблокам (heat map)
- Время чтения фблока (полного и частичного)
- Скорость чтения (байт/сек)
- Доля частичных чтений

**Метрики состояния диска:**
- Общее количество фблоков
- Количество и процент битых фблоков
- Количество и процент защищённых фблоков
- Динамика появления новых битых фблоков

**Метрики очереди записи:**
- Текущий размер очереди
- Максимальный размер очереди (пики)
- Скорость наполнения очереди
- Скорость опустошения очереди
- Количество срабатываний backpressure
- Время ожидания в очереди для каждого фблока
- **Время между получением буфера и завершением записи**

#### 7.9.3. Частота сбора
- Операционные метрики (время записи, попытки) — на каждую операцию.
- Метрики состояния — с настраиваемым интервалом (по умолчанию 30 секунд).
- Счётчики скорости — скользящее окно (последние 5-60 секунд).

#### 7.9.4. Доступ к метрикам
Библиотека должна предоставлять **программный доступ** ко всем собираемым метрикам.

Что именно делает библиотека:
- Хранит актуальные значения метрик в памяти.
- Предоставляет API (функции/методы) для их получения.
- Потребитель сам решает, что с ними делать:
  - Вывести в лог.
  - Отправить в Prometheus (через свой экспортёр).
  - Записать в файл.
  - Игнорировать.

**Библиотека не должна**:
- Навязывать конкретный способ экспорта (HTTP, StatsD, ...).
- Требовать внешних зависимостей для работы с метриками.
- Писать метрики на диск (кроме критических событий).

#### 7.9.5. Логирование критических событий
На диск (в отдельный лог-файл) должны писаться:
- Обнаружение битых фблоков.
- Превышение порога битых фблоков (алерт).
- События backpressure (начало/конец).
- Ошибки ввода-вывода.
- Старт/останов системы.

Формат лога — структурированный (JSON) для автоматической обработки.

#### 7.9.6. Настраиваемость
В конфиге должны быть параметры:
- Включить/выключить детальный сбор.
- Интервалы сбора.
- Адреса для экспорта (если используется StatsD).
- Пороги для алертов.

## 8. Восстановление данных

### 8.1. Необходимость восстановления
Поскольку система работает с большими объёмами данных (10-20 ТБ) и возможны сбои:
- Повреждение метаданных раздела.
- Сбой питания в момент записи.
- Физическое повреждение диска (битые сектора).
- Ошибки в работе самой программы.

Необходима возможность **извлекать данные** из раздела в различных сценариях восстановления.

### 8.2. Принцип восстанавливаемости
Благодаря тому, что **UUID хранится в каждом фблоке**, система может:
- Идентифицировать фблоки даже при полной потере метаданных.
- Извлекать отдельные фблоки по UUID.
- Сканировать весь раздел и собирать все найденные фблоки.

### 8.3. CLI программа для восстановления
В состав системы должна входить отдельная программа (command-line interface), которая умеет читать раздел и извлекать данные.

**Основные возможности программы:**

#### 8.3.1. Чтение всего раздела
Программа должна уметь последовательно читать весь файл/раздел, включая:
- Корректные фблоки (с валидным заголовком и контрольной суммой).
- Частично повреждённые фблоки (где некоторые фчанки не читаются).
- Фблоки с битыми секторами (пропуская их, но фиксируя проблему).

#### 8.3.2. Извлечение данных
Для каждого найденного фблока программа должна предоставлять возможность:
- Сохранить данные в отдельный файл.
- Отправить по сети (на другой сервер).
- Записать в другое устройство (как новый фблок или как сырые данные).
- Вывести в поток (stdout) для передачи другой программе.

Формат сохранения/передачи должен быть гибким и определяться пользователем.

#### 8.3.3. Работа с повреждёнными фблоками
- Если фблок повреждён частично, программа должна извлечь те данные, которые можно прочитать.
- Должна быть возможность получить отчёт о том, какие фчанки повреждены.
- Программа не должна падать при encountering битых секторов — только сообщать об ошибках и продолжать.

#### 8.3.4. Поиск по UUID
Программа должна уметь находить фблок по UUID (сканированием раздел, без использования метаданных) и извлекать его.

### 8.4. Независимость от основного приложения
- Программа восстановления должна работать **полностью независимо** от основного приложения.
- Она не должна полагаться на метаданные, созданные основной системой (хотя может их использовать, если они доступны).
- Должна запускаться на любой машине, имеющей доступ к устройству/файлу.

### 8.5. Форматы вывода (на усмотрение реализации)
Программа может поддерживать различные форматы вывода, например:
- Сырые данные фблока.
- Данные с служебной информацией (заголовок + данные).
- Поток фблоков с разделителями.
- Индексный файл со списком найденных UUID и их смещений.

Конкретные форматы будут определены при проектировании программы.

### 8.6. Сценарии использования (примеры)

- Полная потеря метаданных — сканирование всего диска и сохранение всех найденных фблоков.
- Извлечение конкретного фблока по известному UUID.
- Перенос данных с умирающего диска на новый.

### 8.7. Требования к производительности восстановления

- Программа должна работать с дисками 20 ТБ за приемлемое время.
- Желательна поддержка параллельного чтения (если устройство позволяет).
- Должен отображаться прогресс и оценка времени до завершения.

### 8.8. Логирование и отчётность

- Программа должна вести лог всех операций.
- Должна формировать отчёт о найденных фблоках, пропущенных секторах, ошибках чтения.
- Отчёт должен быть в машиночитаемом формате (JSON, CSV) для дальнейшей автоматической обработки.
