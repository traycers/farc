# Хранилище (storage)

## 1. Базовые параметры

Параметры задаются при инициализации хранилища и фиксируются в суперблоке:

| Параметр | Значение | Примечание |
|----------|----------|------------|
| Размер фблока | 512 МБ (536,870,912 байт) | Может быть изменён, но фиксируется при инициализации |
| Размер фчанка | 16 МБ | Для HDD, настраивается при инициализации |
| Размер хранилища | до 20 ТБ и более | Определяется размером диска/файла |
| Количество фблоков | ~37,252 для 20 ТБ | Рассчитывается как размер_хранилища / размер_фблока |

## 2. Суперблок

Расположен в самом начале диска (первый сектор, смещение 0). Размер: 4 КБ (один сектор).

**Структура суперблока:**

| Поле | Размер | Описание |
|------|--------|----------|
| Магическое число | 8 байт | "FBLOCK01" (идентификация хранилища) |
| Версия формата хранилища | 2 байта | Текущая версия = 1 |
| Флаг состояния | 1 байт | 0x01 - чистое завершение, 0x02 - было грязное |
| Резерв | 5 байт | Выравнивание |
| Размер фблока | 8 байт | В байтах |
| Размер фчанка | 8 байт | В байтах |
| Общее количество фблоков | 8 байт | Всего фблоков в хранилище |
| Размер одной записи индекса | 2 байта | = 40 байт |
| Количество копий индексов | 2 байта | Текущее количество копий (минимум 2) |
| Список индексных фблоков | 100 × 8 байт | Массив индексов фблоков, где хранятся копии индексов (максимум 100 копий) |
| Активная копия индексов | 8 байт | Индекс фблока с самой свежей копией |
| Время последнего обновления | 8 байт | Когда обновлялся суперблок |
| Контрольная сумма | 8 байт | xxHash64 всего суперблока |
| Резерв | до 4096 байт | |

**Примечание:** Суперблок обновляется только при:
- Инициализации хранилища
- Расширении хранилища (добавлении новых индексных фблоков)
- Чистом завершении работы

## 3. Фблок

### 3.1. Заголовок фблока

Размер заголовка: 128 байт (фиксированно).

| Поле | Размер | Описание |
|------|--------|----------|
| Магическое число | 4 байта | 0xFB1001 (идентификация фблока) |
| Версия формата фблока | 1 байт | Структура заголовка (текущая = 1) |
| Флаги | 1 байт | 4 флага (см. раздел 3.2) |
| Тип фблока | 1 байт | 0x00 - данные, 0x01 - индексы |
| Резерв | 3 байта | Выравнивание |
| UUID | 16 байт | Уникальный идентификатор |
| Время окончания актуальности | 8 байт | Unix nanoseconds (задаётся потребителем) |
| Время записи | 8 байт | Unix nanoseconds (заполняется при записи) |
| Размер полезных данных | 8 байт | В байтах (может быть меньше размера фблока) |
| Контрольная сумма заголовка | 4 байта | CRC32C всего заголовка |
| Контрольная сумма данных | 8 байт | xxHash64 от полезных данных |
| Резерв | 58 байт | Для будущих расширений |

**Итого:** 4+1+1+1+3+16+8+8+8+4+8+58 = 128 байт

### 3.2. Флаги (1 байт)

| Бит | Флаг | Описание |
|-----|------|----------|
| 0 | записан | Фблок содержит данные (UUID валиден) |
| 1 | в процессе | Идёт запись, фблок не завершён |
| 2 | битый | Физическое повреждение, использовать нельзя |
| 3 | заблокирован | Защита от перезаписи (read-only) |
| 4-7 | резерв | Для будущего |

### 3.3. Типы фблоков

- **0x00 — данные потребителя:** обычный фблок с полезными данными
- **0x01 — индексы:** фблок содержит копию индексов всего хранилища

UUID для индексных фблоков имеет специальный формат:
- Младшие 8 байт = 0
- Старшие 8 байт = номер копии (1, 2, 3...)

Пример: `00000000-0000-0000-0000-000000000001` для первой копии.

## 4. Индексы (метаданные)

### 4.1. Формат записи индекса

Каждая запись соответствует одному фблоку (по его индексу). Размер: 40 байт.

| Поле | Размер | Описание |
|------|--------|----------|
| UUID | 16 байт | UUID фблока (0 = свободен) |
| Время окончания актуальности | 8 байт | Unix nanoseconds |
| Время записи | 8 байт | Unix nanoseconds |
| Флаги | 1 байт | Те же 4 флага, что в заголовке |
| Версия индексов | 1 байт | Версия структуры индексов |
| Резерв | 2 байта | Выравнивание |
| Контрольная сумма записи | 4 байта | CRC32C этой записи |

**Итого:** 16+8+8+1+1+2+4 = 40 байт

### 4.2. Хранение индексов

Индексы хранятся в **обычных фблоках** с типом `0x01`.

**Количество копий:**
- Минимум 2 копии (в начале и в конце диска)
- Рекомендуется 3 копии (начало, середина, конец) для большей надёжности
- При расширении создаются новые копии в новом конце

**Расположение при инициализации:**
- Первые K фблоков: копии индексов (K = количество копий)
- Последние K фблоков: ещё K копий индексов
- Остальные фблоки: данные потребителя

**Пример для 10,000 фблоков и 2 копий:**
- Фблок 0: индексы (копия 1)
- Фблок 1: индексы (копия 2)
- Фблоки 2-9997: данные
- Фблок 9998: индексы (копия 3)
- Фблок 9999: индексы (копия 4)

### 4.3. Как найти все копии при старте

1. Читаем суперблок — там список всех индексных фблоков
2. Читаем все копии индексов из этих фблоков (каждая копия читается как обычные данные из фблока)
3. Сравниваем время записи каждой копии (хранится в начале данных индексов)
4. Берём самую свежую целую копию (с совпадающей контрольной суммой)

Время: количество_копий × (seek + чтение 400-500 КБ) = **~40-60 мс**

## 5. Расширение хранилища

Расширение происходит без остановки записи.

**Исходное состояние:**
- Размер S, N фблоков
- Индексные фблоки: в начале и в конце

**Шаг 1: Добавление места**
- Увеличиваем размер диска/файла до S_new (S_new > S)
- Появляются новые фблоки с индексами от N до N_new-1

**Шаг 2: Создание новых индексных фблоков**
- Выделяем M фблоков в новом конце (M = количество копий)
- Записываем в них текущие индексы (копируем из памяти)
- Помечаем их тип = 0x01, UUID = специальный

**Шаг 3: Обновление суперблока**
- Добавляем новые индексные фблоки в список
- Обновляем поле "активная копия" (указываем на самую свежую)
- Записываем суперблок

**Шаг 4: Продолжение работы**
- Старые индексные фблоки остаются на месте
- Они будут перезаписаны данными при циклической записи (если окажутся в зоне данных)
- Или останутся как исторические копии (занимают мало места)

**Время расширения:** только на запись индексов в новые фблоки + обновление суперблока = миллисекунды.

## 6. Процесс записи фблока (с точки зрения хранилища)

1. **Выбор целевого фблока:**
   - Циклически обходим фблоки, начиная с последнего записанного
   - Пропускаем битые и заблокированные
   - Если фблок записан, проверяем `текущее_время >= время_окончания_актуальности`

2. **Пометка "в процессе":**
   - В заголовке целевого фблока выставляем флаг `в процессе = 1`
   - Записываем заголовок (только первую копию? или все?)

3. **Запись по фчанкам с verify:**
   - Для каждого фчанка:
     - Запись на диск
     - Чтение обратно
     - Сравнение с оригиналом
     - При несовпадении — повтор (до 3 раз)
     - Если всё равно не совпадает — весь фблок считается битым

4. **При успехе:**
   - Генерируем UUID
   - Заполняем заголовок: UUID, время_записи, время_окончания, размер_данных
   - Снимаем флаг `в процессе`, выставляем флаг `записан`
   - Записываем финальный заголовок

5. **Обновление индексов:**
   - В памяти: обновляем запись для этого индекса (UUID, время_окончания, время_записи, флаги)
   - Для каждой копии индексов (из списка в суперблоке):
     - Читаем фблок с индексами
     - Обновляем в нём нужную запись (по индексу фблока)
     - Пересчитываем контрольную сумму
     - Записываем фблок целиком (с verify)

6. **Уведомление потребителя:**
   - Отправляем событие о новом фблоке (UUID, время_записи)

**Время записи фблока:** ~25-30 секунд (для 512 МБ с verify)

## 7. Восстановление после сбоя

При старте системы:

1. **Читаем суперблок** (первый сектор)
2. **Проверяем контрольную сумму** суперблока
3. **Получаем список индексных фблоков** из суперблока
4. **Читаем все копии индексов:**
   - Для каждого индексного фблока читаем его данные
   - Проверяем контрольную сумму всей копии
   - Запоминаем время записи копии (хранится в начале данных)
5. **Выбираем самую свежую целую копию** (по времени записи)
6. **Загружаем индексы в память**
7. **Сканируем фблоки с флагом "в процессе":**
   - Читаем заголовки всех фблоков (быстрое сканирование 128-байтовых заголовков)
   - Находим фблоки с флагом `в процессе = 1`
   - Помечаем их в индексах как `битые`
   - Обновляем копии индексов (записываем в индексные фблоки)

Время восстановления:
- Чтение суперблока: 1 seek
- Чтение копий индексов: 2-3 seeks + 1-2 МБ данных = **~40 мс**
- Сканирование заголовков: 37,000 × 128 байт = 4.7 МБ, но с seek-ами = **~2-3 минуты** (можно делать в фоне)

**Оптимизация:** сканирование заголовков для поиска "в процессе" можно делать в фоновом потоке, а основную работу начинать сразу после загрузки индексов.

## 8. Размещение на диске (пример для 20 ТБ)

Параметры:
- Размер фблока: 512 МБ
- Количество фблоков: 37,252
- Количество копий индексов: 4 (2 в начале, 2 в конце)

**Объём под индексы:** 4 × 512 МБ = 2 ГБ (0.01% от 20 ТБ)
