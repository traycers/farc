# Документация проекта: Работа с дисковыми разделами в Go

## 1. Цель и требования

Мы пишем приложение, которое работает с дисковыми разделами (D:, E:, F: на Windows, /dev/sdb1 на Linux) как с «сырым» пространством. Весь раздел занят одним большим файлом, с которым работает только наша программа. В каждый момент времени возможна только одна операция: либо чтение, либо запись.

### Ключевые требования:
1. **Производительность на Linux** — максимальная скорость ввода-вывода.
2. **Переносимость файлов** — файл, созданный на Linux, должен корректно читаться на Windows (и наоборот).
3. **Фиксированные блоки** — чтение и запись большими блоками (например, 400 МБ или 2 ГБ).
4. **Эксклюзивный доступ** — никакой другой процесс не должен трогать раздел.

---

## 2. Почему мы выбрали такой подход

### Критические решения:
- **Linux**: Используем `O_DIRECT`, чтобы данные шли напрямую на устройство, минуя кэш ядра. Это даёт предсказуемую производительность и контроль.
- **Windows**: Используем обычный `os.File` (с флагом `os.O_SYNC`). Производительность не критична, главное — совместимость формата.
- **Единый формат данных**: Все числа записываются в little-endian, выравнивание — по 4 КБ (безопасно для всех платформ).
- **Большие блоки**: Работа блоками по 400 МБ или 2 ГБ минимизирует количество системных вызовов и идеально подходит для `O_DIRECT`.

### Что мы отвергли:
- **io_uring** — избыточен для нашего сценария (один поток, один файл, последовательные операции).
- **Буферизованный ввод-вывод** — не даёт гарантий записи и контроля.
- **Сложные форматы данных** — пока не нужны; начинаем с простой адресации блоков.

---

## 3. Как это работает на Linux (O_DIRECT)

### Открытие устройства
- Открываем само блочное устройство (например, `/dev/sdb1`), а не точку монтирования.
- Флаги: `O_RDWR | O_DIRECT | O_SYNC`.
  - `O_DIRECT` — прямой ввод-вывод, без кэширования в ядре.
  - `O_SYNC` — каждая запись синхронная (ждём, пока данные уйдут на диск). Можно заменить на ручной `Sync()` для гибкости.

### Требования к данным
- **Выравнивание**: Буфер должен быть выровнен по границе сектора (обычно 512 байт, безопасно — 4 КБ).
- **Размер**: Длина данных должна быть кратна 512 байт (или 4 КБ).
- **Память**: Для больших блоков (2 ГБ) используем `mmap`, чтобы гарантировать выравнивание и избежать давления на GC.

### Гарантия записи
- Вызов `Sync()` после критических операций (или полагаемся на `O_SYNC`).

---

## 4. Как это работает на Windows

### Открытие файла
- Используем `os.OpenFile` с флагами `os.O_RDWR | os.O_CREATE | os.O_SYNC`.
- `os.O_SYNC` в Go на Windows транслируется в `FlushFileBuffers` после каждой записи.
- Никаких требований к выравниванию — обычный файл.

### Гарантия записи
- Вызываем `Sync()` после каждой логической операции, чтобы данные точно попали на диск.

---

## 5. Единый формат данных (для переносимости)

Чтобы файл можно было копировать между Linux и Windows, мы договорились:

### Структура на диске
1. **Блок 0** — заголовок раздела (метаданные).
2. **Блоки 1..N** — битовая карта свободных/занятых блоков.
3. **Остальные блоки** — пользовательские данные.

### Правила сериализации
- Все многобайтовые числа — **little-endian** (общий порядок для x86/x64).
- Размер всех полей фиксирован (никакой `encoding/gob`, только ручная упаковка).
- Выравнивание — по границе 4 КБ (это размер кластера NTFS и блока ext4 по умолчанию).

### Пример заголовка (блок 0, упрощённо)
```
offset 0:  [8]byte  Magic            "MYFS2026"
offset 8:  uint32   Version          1
offset 12: uint64   BlockSize        4294967296 (4 ГБ, если решим)
offset 20: uint64   TotalBlocks      1000
offset 28: uint64   BitmapBlocks      16
offset 36: uint32   Checksum         0x12345678
```

## 6. Архитектура кода (Go)

### Платформозависимые реализации

**Интерфейс `Partition`** (общий для всех):
```go
type Partition interface {
    ReadAt(p []byte, off int64) (int, error)
    WriteAt(p []byte, off int64) (int, error)
    Sync() error
    Close() error
}
```

**Linux** (`partition_linux.go`):
- Открытие с `unix.O_DIRECT`.
- Проверка выравнивания.
- Выделение выровненных буферов.
- Условная компиляция: `//go:build linux`.

**Windows** (`partition_windows.go`):
- Открытие через `os.OpenFile` с `os.O_SYNC`.
- Без проверок выравнивания.
- Условная компиляция: `//go:build windows`.

### Работа с блоками
- Функции `WriteBlock(index int64, data []byte)` и `ReadBlock(...)`.
- Внутри — проверка, что `len(data) == blockSize`.
- Для очень больших блоков (2 ГБ) можно добавить запись с прогрессом (разбивку на чанки), чтобы не «вешать» интерфейс.

## 7. Важные ограничения и предупреждения

### Linux
- **Права доступа**: Программа должна запускаться от root или пользователь должен быть в группе `disk`.
- **Раздел не должен быть смонтирован**: Иначе файловая система может конфликтовать с прямыми записями.
- **Выравнивание**: Любая ошибка выравнивания вернёт `EINVAL` без понятного сообщения. Проверяем всё в коде.

### Windows
- **Производительность**: Может быть ниже, но это осознанный компромисс.
- **Имена устройств**: Работаем с обычными файлами, а не с `\\.\PhysicalDriveN` (пока не требуется).

### Переносимость
- Файлы можно копировать любой утилитой (`cp`, `robocopy`, проводник).
- Важно, чтобы приложение на целевой платформе использовало **тот же размер блока**, что и при записи (иначе адресация съедет).

---

## 8. Что мы не стали делать (и почему)

1. **io_uring** — избыточно для одного файла и последовательного доступа.
2. **Асинхронный ввод-вывод на Windows** — усложнение без выгоды (производительность не важна).
3. **Буферизация в Go** — `os.File` не буферизирован, а `bufio` дал бы меньше контроля.
4. **Поддержка разреженных файлов** — пока не нужно; весь раздел выделен целиком.

---

## 9. Типичный сценарий использования

```go
// Linux
part, _ := OpenPartition("/dev/sdb1", 2*1024*1024*1024)
defer part.Close()

buf := make([]byte, part.BlockSize())
// ... заполняем buf данными

part.WriteBlock(42, buf)   // пишем блок №42
part.Sync()                 // гарантируем запись
```

```go
// Windows
part, _ := OpenPartition("D:\\data.bin", 2*1024*1024*1024)
// ... то же самое
```
